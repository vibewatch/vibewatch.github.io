## 1. 人们在讨论什么

### 1.1 Thin Harness Doctrine 成形 🡕

昨天关于 harness engineering 的讨论，今天沉淀成一条具体的架构原则。[@garrytan](https://x.com/garrytan) 发布了他称为 agentic engineering [“最简单提炼”](https://x.com/garrytan/status/2043566215927328955)的三层栈：顶层是 fat skills（用 markdown procedures 编码 judgment 和 domain knowledge），中间是 thin CLI harness（约 200 行代码，JSON in、text out，默认只读），底层是 deterministic application code（QueryDB、ReadDoc、Search）。原则的方向很明确：“把 intelligence 上推到 skills，把 execution 下推到 deterministic tooling。保持 harness thin。”

![三层 agent stack 架构文字：顶层 fat skills，中间 thin CLI harness，底层 deterministic application](https://pbs.twimg.com/media/HFw0oV5XkAAdj_U.jpg)

这条推文获得 1,299 个点赞和 1,666 个收藏。[@MindTheGapMTG](https://x.com/MindTheGapMTG) 的回复补充了一个操作细节：“Fat skills 不是写出来的，而是积累出来的。我们 CLAUDE.md 里的每一行，都是一次生产故障的事后复盘。” [@weareuplers](https://x.com/weareuplers) 把它联系到 Unix 哲学：“Keep the harness thin 是 agentic engineering 版本的 ‘do one thing well.’ 我们花了 50 年才为 Unix 学会这个教训，而现在似乎正在用智能体快速重演同一课。”

[@DanielMiessler](https://x.com/DanielMiessler) [区分了](https://x.com/DanielMiessler/status/2043488548318597181)两类 harness engineering：(1) 准确告诉系统如何做事，随着模型进步这会“被吃掉”（Bitter Lesson）；(2) 告诉系统什么是好的——解释你是谁、你喜欢什么，以及优秀输出意味着什么。他认为只有第二类是面向未来的。

[@dexhorthy](https://x.com/dexhorthy) [完全反驳了](https://x.com/dexhorthy/status/2043707427632431509)这个框架：“想象一下，如果你一年前就学会了 12-factor context engineering，就能自信跳过所有这些 ‘harness’ 炒作并回去工作。” 这条推文获得 102 个点赞，[@johns10d](https://x.com/johns10d) 给出了实质性回复：“Harness engineering 是在模型周围放置流程性代码，保证它做你想要的事，而不是礼貌地请求它。礼貌请求并不能带我们走完全程。”

### 1.2 Agent Skills 生态达到规模 🡕

Agent skills 生态现在有了具体数字。[@nozmen](https://x.com/nozmen) [上线了 officialskills.sh](https://x.com/nozmen/status/2043658478929985849)，这是一个精选目录，列出 464 个 skills（314 个来自开发团队的官方 skills、150 个社区 skills），覆盖 40 个 vendor organizations 和 11 个 categories。兼容 Claude Code、Codex、Cursor、GitHub Copilot 和 OpenCode。Skills 来自 Microsoft、Anthropic、Google、Sentry、Cloudflare、Trail of Bits 等团队。该站点使用现有的 `npx skills` 命令安装。

### 1.3 Agent Security 从理论走向证据 🡕

智能体安全担忧今天产生了具体研究。[@askalphaxiv](https://x.com/askalphaxiv) [分享](https://x.com/askalphaxiv/status/2043774016293384256)了 “Your Agent Is Mine”（arXiv:2604.08407），这是第一项关于恶意 LLM API routers 的系统研究。在测试的 28 个付费 routers 和 400 个免费 routers 中，研究者发现 1 个付费和 8 个免费 routers 会主动注入恶意代码，2 个部署 adaptive evasion triggers，17 个触碰了研究者拥有的 AWS canary credentials。这些 routers 已处理 21 亿 tokens，暴露了 440 个 codex sessions 中的 99 份 credentials。论文呼吁为 tool calls 提供端到端完整性，让 clients 能验证收到的 action 与 provider 产生的 action 完全一致。

![关于 LLM supply chain 中恶意中介攻击的学术论文首页](https://pbs.twimg.com/media/HFzxmtqWoAAWFhf.png)

[@ZackKorman](https://x.com/ZackKorman) [称 AIUC-1](https://x.com/ZackKorman/status/2043679021834154020)——一个新的 AI agent security 合规标准——是“巨大的骗局”，尽管其技术贡献者名单包括 ElevenLabs、Cisco、OWASP、MITRE、Microsoft、Stanford、Google Cloud、Anthropic、Meta、Databricks 和 Visa。在[后续](https://x.com/ZackKorman/status/2043794863347450214)中，他澄清：“如果有人想做一个 AI agent security framework 并免费发出来，随便。但把它做成 compliance standard” 才是问题。

![AIUC-1 consortium members grid，展示 ElevenLabs、Cisco、OWASP、Microsoft、Anthropic、Meta 等 logo](https://pbs.twimg.com/media/HFyafP_XYAAEPSL.jpg)

[@AllAICoder](https://x.com/AllAICoder) [警告](https://x.com/AllAICoder/status/2043643043497062722)，从开放 marketplace 使用第三方 skills 时，“一个可疑 skill 就能接管你的整台机器”。[@RogoAI](https://x.com/RogoAI) [报告](https://x.com/RogoAI/status/2043735515917169015)，他们的 autonomous pen-testing agent Sisyphus “一个下午发现了 18 个 manual testing 漏掉的可利用问题。”

### 1.5 Context Engineering 与 Agent Memory 🡒

Context engineering 继续作为稳定主题出现，并带来新的视觉 taxonomy 和 memory solutions。[@DataScienceDojo](https://x.com/DataScienceDojo) [发布](https://x.com/DataScienceDojo/status/2043781280890749318)了一张信息图，定义 context engineering 的 6 个组件：Instructions/System Prompt、Long-Term Memory、Available Tools、Structured Output、User Prompt 和 Retrieved Information (RAG)。

![Context engineering 信息图，展示 instructions、long-term memory、available tools、structured output、user prompt、RAG 六个组件](https://pbs.twimg.com/media/HFz4PHkXAAAPJgS.jpg)

[@ghumare64](https://x.com/ghumare64) [推荐](https://x.com/ghumare64/status/2043661245635493920) agentmemory 作为跨 agent memory layer，可跨 harnesses 工作，拥有 95.2% retrieval R@5、减少 92% tokens、43 个 MCP tools 和 654 个 passing tests。[@unbrowse](https://x.com/unbrowse) [提出](https://x.com/unbrowse/status/2043690768406290670)了反向方法：“如果你就是不复制数据呢？每个 agent memory system 都把数据复制到 vector store。要是 agent 直接索引源数据呢？”

[@che_shr_cat](https://x.com/che_shr_cat) [分享](https://x.com/che_shr_cat/status/2043759724114395460)了 Memory Intelligence Agent 论文（arXiv:2604.04503），其中一个 7B-parameter agent 采用 Manager-Planner-Executor memory architecture，通过把 procedural memory 与 execution 解耦，并在 inference 期间更新 weights，表现比 32B model 高 18%。

![Memory Intelligence Agent 论文，展示 Manager-Planner-Executor 架构以 7B model 实现 31% 改进](https://pbs.twimg.com/media/HFzjhn1XQAAnwFN.jpg)

## 2. 令人困扰的问题

### Agent Harness 资源消耗（High）

[@0xClandestine](https://x.com/0xClandestine) [报告](https://x.com/0xClandestine/status/2043748820765290535)，一个 opencode agent session 会消耗 5GB RAM，称其“不可接受”，并请求 Rust 或 Zig 编写的替代品。系统监控截图确认，在一台 64GB RAM 的 Apple M4 Max 上，单个 opencode 进程占用 509.8MB resident / 4.8GB virtual memory。回复中没有提出令人满意的替代方案。

![系统监控显示 opencode process 在 Apple M4 Max 上消耗 4.8GB virtual memory](https://pbs.twimg.com/media/HFzac-8asAU4eyP.jpg)

### Coding Agents 中的 Subagent 可见性（Medium）

[@dani_avila7](https://x.com/dani_avila7) [指出](https://x.com/dani_avila7/status/2043749681633587499)，Claude Code 中存在一个具体 UX 问题：“使用会调用 subagents 的 Skills 时，subagent 不会显示在 Claude Code interface 中。一切都能工作，但你看不出实际是你在 skill frontmatter 中添加的 subagent，还是内置 agent 在工作。” 截图展示了通过 frontmatter fields 进行 skill-to-subagent linking 的机制，但 TUI 层面没有提示。

### Multi-Agent 需求导致 Token Shortage（Medium）

[@Grummz](https://x.com/Grummz) [警告](https://x.com/Grummz/status/2043749800089399329)：“我们正走向 token shortage。不只是 compute limits，每个人对 AI 的需求都爆炸了。而且现在全是 multi-agent。” 在回复中，他量化说：“现在每个 AI harness 每人会跑 4-8x 次 LLM calls，而不是 1 次。”

### Enterprise Agent Orchestration 怀疑（Medium）

[@buccocapital](https://x.com/buccocapital) [嘲讽](https://x.com/buccocapital/status/2043682978430103777)企业 SaaS 公司声称自己是“管理 agent access、security 和 orchestration 的 neutral party”，获得 179 个点赞。回复进一步尖锐化批评：[@curtismakes](https://x.com/curtismakes) 观察到“每家 SaaS 公司都在同一个季度从 AI-powered 转向 AI-orchestrator”，而 [@sigmadeltacto](https://x.com/sigmadeltacto) 预测现实会是：“neutral party: $500k ACV, 3-year lock-in, Professional Services required.”

### Voice Agent Services Scams（Medium）

[@huzzymad](https://x.com/huzzymad) [报告](https://x.com/huzzymad/status/2043740316344168919)，一位家人因为一个 voice agent demo 被骗 6,000 美元，且“没有 logs、没有 transcripts、零 infrastructure ownership、每周 calls 有限（额外 calls 另收费）”。随着 voice agent market 增长，这种卖昂贵 demo 但不交付生产基础设施的模式似乎正在出现。

### MCP Overuse（Low）

[@jezell](https://x.com/jezell) [认为](https://x.com/jezell/status/2043417554535010337)，MCP 被过度使用：“如果 agent 和 backend services 的代码都由你控制，99.9% 的时间你都不应该用 MCP。MCP 解决了一个真实问题：为别人家的东西构建 connector marketplace。” 所有 LLMs 都支持直接 tool calls，不需要 MCP。

## 3. 人们期望的功能

### Self-Updating Skills

[@avisinghdotdev](https://x.com/avisinghdotdev) [请求](https://x.com/avisinghdotdev/status/2043651542813921674) Claude Code 提供 `/update-skills` 命令，类似现有的 `/create-skills`，这样“agent can update the existing skill based on past interaction.” 如今 skills 是静态 artifacts；没有机制让 skills 从使用中演化。

### Multi-Agent Windowed Workflows

[@Kraggich](https://x.com/Kraggich) [指出](https://x.com/Kraggich/status/2043814179430162696)一个核心 UX 缺口：“每个 AI coding tool 都在犯同一个错误。Cursor、Windsurf、Codex、Claude Code——它们都给你一个窗口里的一个 agent。但真实工作不是一次一个任务。它是三个 agents 在三个 worktrees 中解决同一个问题的三个部分。”

### Enterprise Voice Agent Testing

[@sumanyu](https://x.com/sumanyu) [预测](https://x.com/sumanyu/status/2043718422198927593)，“每家 YC W26 voice agent 公司都会面临同一个企业问题：‘你们怎么测试它？’ 不是你的 demo。不是你的 benchmark。你怎么用我们的数据、我们的 edge cases、我们的 compliance requirements、以我们的规模测试它？”

### Lightweight Coding Agent Harnesses

[@0xClandestine](https://x.com/0xClandestine) [要求](https://x.com/0xClandestine/status/2043748820765290535)一个 RAM 消耗更低的 coding agent harness，“最好用 Rust/Zig 写。” 回复中没有令人满意的答案。现有工具（消耗数 GB 的 Node.js/Python harnesses）与从业者想要的东西（轻量 native harnesses）之间的缺口仍未填补。

### Agent Governance for Teams

[@PestoPoppa](https://x.com/PestoPoppa) [开源](https://x.com/PestoPoppa/status/2043626137520116056)了一个用于协作 agent workflows 的治理层：“如果你的开发者在用 Claude Code / Codex，但 sessions 不能互相继承、知识会蒸发、onboarding 很痛苦，这个 repo 就是给你的。” 底层需求是 team-level agent coordination，且能跨 sessions 持久存在。

## 6. 新动态与亮点

### 首个标准化 Voice Agent Benchmark

[@tulseedoshi](https://x.com/tulseedoshi) [分享](https://x.com/tulseedoshi/status/2043705774308733131)了 Sierra Platform 的 tau-voice leaderboard，这是首个面向 real-time voice agent performance 的标准化 benchmark。当前排名：Gemini 3.1 Flash Live (Thinking) 为 43.8% PASS，xai-realtime 为 38.3%，gpt-realtime-1.5 为 35.3%，gemini-live-2.5-flash-native-audio 为 25.8%。类别包括 retail、airline 和 telecom scenarios。

![tau-voice leaderboard，显示 Gemini 3.1 Flash Live 43.8%、xai-realtime 38.3%、gpt-realtime-1.5 35.3%](https://pbs.twimg.com/media/HFwqB4NaUAAsEjQ.jpg)

### 更好的 Memory 让小型 Agent Models 击败大型模型

Memory Intelligence Agent 论文（arXiv:2604.04503）证明，一个带 Manager-Planner-Executor memory architecture 的 7B-parameter model 实现了 31% 平均提升，并在评估数据集上比 32B model 高 18%。关键技术是：把 procedural memory 与 execution 解耦，并通过 parametric 与 non-parametric memory 之间的双向转换，让 memory 在 inference 期间演化。

### Agents 自主发现并订阅 APIs

[@OrbisAPI](https://x.com/OrbisAPI) [报告](https://x.com/OrbisAPI/status/2043766007081160833)，Claude agents 正在自主发现 Orbis API catalog、浏览可用 services、注册并订阅 APIs，无需人工干预。凭借 730+ APIs 和 x402 micropayments，这代表了 emergent agent economy behavior 的早期证据。[@grok](https://x.com/grok) 在 Lightning Network 上[描述](https://x.com/grok/status/2043820017905496098)了一个相关模式：“AI agent 能在几秒内启动自己的 L402 server……另一个 agent 发现它、用 sats 即时付款、证明 preimage，并消费服务。Zero setup、zero KYC、完全 machine-to-machine。”

### Microsoft Agent Framework 达到 1.0

## 7. 机会在哪里

**[+++] 强信号：Agent Security Tooling and Verification。** “Your Agent Is Mine” 论文记录了影响数十亿 tokens 的 LLM API routers 主动攻击。[@AllAICoder](https://x.com/AllAICoder) 警告 malicious skills。[@ZackKorman](https://x.com/ZackKorman) 批评 AIUC-1 是过早的 compliance theater。已记录威胁和可用防御之间的缺口很大。End-to-end tool call integrity、skill verification 和 transparent router auditing 都是即时需求。（[source](https://x.com/askalphaxiv/status/2043774016293384256)）

**[+++] 强信号：Skill Quality and Lifecycle Management。** 生态现在已有来自 40 个 vendor teams 的 464 个 catalogued skills，但没有机制让 skills 从使用中演化。[@avisinghdotdev](https://x.com/avisinghdotdev) 请求 `/update-skills`；[@_philschmid](https://x.com/_philschmid) 发布了 retiring skills 的指导。Skill lifecycle——creation、evaluation、improvement、retirement——仍完全手动。一个能追踪 skill effectiveness 并自动化改进的系统，将解决不断增长的痛点。（[source](https://x.com/avisinghdotdev/status/2043651542813921674)）

**[++] 中等信号：Lightweight Native Agent Harnesses。** 当前 Node.js 和 Python 写的 coding agent harnesses 会消耗数 GB RAM。[@0xClandestine](https://x.com/0xClandestine) 记录了单个 opencode session 占用 5GB，并要求 Rust/Zig 替代品。[@a_g_e_n_c](https://x.com/a_g_e_n_c) 展示了一个运行在 512MB RAM 上的完整 agent runtime。臃肿主流 harnesses 与极简硬件可支持能力之间的差距，为 native、resource-efficient agent runtimes 创造了机会。（[source](https://x.com/0xClandestine/status/2043748820765290535)）

**[++] 中等信号：Enterprise Voice Agent Testing Infrastructure。** [@sumanyu](https://x.com/sumanyu) 指出了关键企业 blocker：用客户数据、edge cases 和 compliance requirements 测试 voice agents。Sierra 的 tau-voice benchmark 提供了标准化评估，但还没有面向企业特定 voice agent validation 的工具。[@huzzymad](https://x.com/huzzymad) 记录了 voice agent services market 中 6,000 美元骗局，说明需求增长快于质量保证。（[source](https://x.com/sumanyu/status/2043718422198927593)）

**[++] 中等信号：Agent-to-Agent Commerce Protocols。** Orbis（x402 micropayments）、MoonPay CLI（3M tool calls）、Lightning/L402 和多个 DeFi integration layers 都在朝 machine-to-machine payments 构建。基础设施碎片化，但模式一致：agents 需要在无人干预下向其他 agents 购买服务。第一个取得 meaningful network effects 的 protocol，将定义 agent economy 的支付 rail。（[source](https://x.com/OrbisAPI/status/2043766007081160833)）

**[+] 新兴信号：Team Agent Governance。** [@PestoPoppa](https://x.com/PestoPoppa) 开源了协作 agent workflows 的 governance layer，[@nyk_builderz](https://x.com/nyk_builderz) 的 mission-control 达到 4,000+ stars。问题——sessions 不能互相继承、知识蒸发、onboarding 痛苦——对任何使用多个 coding agents 的团队都是真实存在的。（[source](https://x.com/PestoPoppa/status/2043626137520116056)）

**[+] 新兴信号：Agent Memory Architecture Innovation。** Memory Intelligence Agent 论文显示，memory architecture 对 agent performance 的重要性超过 model scale。[@unbrowse](https://x.com/unbrowse) 提出原地索引源数据，而不是复制到 vector stores。agentmemory 提供 95.2% retrieval 的跨工具 memory。主流 RAG-to-vector-store 模式可能会被保留数据原位、使用更智能 retrieval strategies 的架构取代。（[source](https://x.com/che_shr_cat/status/2043759724114395460)）

## 8. 要点总结

1. **“Thin harness, fat skills” 架构已经巩固为主导 agent design principle。** Garry Tan 的提炼——把 intelligence 推入 markdown skills、把 execution 推入 deterministic code、保持 harness minimal——获得 1,299 个点赞和 1,666 个收藏。多位从业者独立验证了这一模式。（[source](https://x.com/garrytan/status/2043566215927328955)）

2. **Agent skills 生态现在有了具体规模：来自 40 个 vendor teams 的 464 个 catalogued skills，分发基础设施已就位，但缺少 lifecycle management。** officialskills.sh 发布精选目录；Google 发布 6 阶段 skills pipeline；Arcium 和 MiniMax 发布 domain-specific skills。缺失环节是能从使用中演化，而不是需要手动维护的 skills。（[source](https://x.com/nozmen/status/2043658478929985849)）

3. **Agent security threats 是已记录且活跃的，不是理论。** “Your Agent Is Mine” 论文发现付费 API routers 中的 malicious code injection、影响数十亿 tokens 的 credential exfiltration，以及野外的 adaptive evasion techniques。与此同时，AIUC-1 的 compliance standard 遭到尖锐批评。这个领域需要可工作的防御，而不是治理剧场。（[source](https://x.com/askalphaxiv/status/2043774016293384256)）

4. **Agent marketplaces 正在从 listings 转向 live commerce，agents 可以自主发现并支付服务。** Orbis 报告 agents 独立浏览、注册并订阅 APIs。MoonPay CLI 达到 300 万次 tool calls。Agent economy 不再是概念；它正在产生可衡量的交易量。（[source](https://x.com/OrbisAPI/status/2043766007081160833)）

5. **Memory architecture 对 agent performance 的重要性超过 model scale。** 一个带专门 memory 的 7B agent 表现比 32B model 高 18%。agentmemory 展示了 95.2% retrieval 和跨工具兼容。Agent 质量的竞争优势正在从 model selection 转向 memory engineering。（[source](https://x.com/che_shr_cat/status/2043759724114395460)）

6. **Voice agents 有了第一个标准化 benchmark，但 enterprise testing 仍是未解决问题。** tau-voice leaderboard 将 Gemini 3.1 Flash Live 排在 43.8%，建立了基线。但企业买家需要用自己的数据和合规要求测试，而 voice agent services market 已经出现 scams。（[source](https://x.com/tulseedoshi/status/2043705774308733131)）

7. **Harness engineering 术语争论是有生产力的，不只是语义之争。** Miessler 区分了 future-proof harness engineering（什么是好）与 fragile harness engineering（如何做）。Dexhorthy 认为 context engineering 已经覆盖这个概念。Johns10d 捍卫了区别：流程性保证不同于礼貌请求。这场争论正在澄清 agent configuration 中真正重要的东西。（[source](https://x.com/DanielMiessler/status/2043488548318597181)）
