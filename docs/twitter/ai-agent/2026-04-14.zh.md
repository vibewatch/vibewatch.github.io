## 1. 人们在讨论什么

### 1.1 Harness Engineering 从教义走向参考架构 🡕

昨天确立了“thin harness, fat skills”教义。今天，社区开始围绕它构建参考基础设施。[@Vtrivedy10](https://x.com/Vtrivedy10) [发布了一篇 first-principles breakdown](https://x.com/Vtrivedy10/status/2043870915059236966)，解释 harnesses 为什么存在：从模型单独无法完成什么反推，模型只是 token input/output machines，需要为 tool access、memory、planning 和 verification loops 进行增强。这条推文获得 206 个点赞和 204 个收藏——是今天数据集中最高分。

![Harness engineering mental model，展示 harnesses 作为模型增强层存在的原因](https://pbs.twimg.com/media/HF1Jw62XQAArH77.jpg)

![Harness engineering diagram，展示 tool access、memory、planning、verification 等组件](https://pbs.twimg.com/media/HF1Jw66XsAAIf2d.jpg)

[@codylindley](https://x.com/codylindley) 在 codylindley.github.io [发布了 harness engineering compatibility matrix](https://x.com/codylindley/status/2043911683757035774)，比较 AGENTS.md、CLAUDE.md、GEMINI.md、copilot-instructions.md 等 instruction formats、自定义 agent systems 和 skill formats 在 Copilot、Claude Code、OpenCode、Gemini CLI、Cursor、Windsurf 等 10+ tools 中的支持情况。该帖 26 个点赞、31 个收藏，是数据集中 bookmark-to-like ratio 最高的帖子，显示从业者对参考资料有强烈需求。

[@loiane](https://x.com/loiane) [发布了一篇博客文章](https://x.com/loiane/status/2044066370002768295)，把 harness engineering 定义为一个 control system，包含两根支柱：feedforward（模型运行前的 guidance——instructions、context、examples、schemas）和 automated feedback（执行后的 sensors——tests、schema validation、business-rule checks）。她引用 Martin Fowler 的 harness engineering 文章，并把这个概念映射到 software delivery lifecycle，认为转变是“from in-the-loop to on-the-loop”。

[@Infoxicador](https://x.com/Infoxicador) [认为](https://x.com/Infoxicador/status/2044045349086474258)，如果 harness engineering 成为常态，modularity 和 tight verification loops 就是应该投资的地方——否则，“tokenmaxxing without reading the code” 会产生脆弱系统。这条推文获得 30,328 次浏览和 34 个收藏。

![展示 modularity 和 close verification loops 是 harness engineering 必要补充的图](https://pbs.twimg.com/media/HF3nA1Oa4AAG56Q.jpg)

**Discussion insight:** [@KSimback](https://x.com/KSimback) [展示](https://x.com/KSimback/status/2044003658166091968)了如何使用个人知识库研究 harness engineering，尤其分析哪些元素构成 moat，哪些会 commoditized。[@SynabunAI](https://x.com/SynabunAI) 的回复提出了关键洞察：“the moat is context that survives sessions. Models are increasingly commoditized. The memory layer is where the lock-in actually is.”

**Comparison to prior day:** 昨天的讨论确立了 thin-harness 原则。今天则分化到实现细节：compatibility matrices、control-system frameworks 和显式 moat analysis。话语从“what is harness engineering?” 转向“how do I actually build one?”

### 1.2 Skills Self-Improvement 成为实践 🡕

今天三支独立团队发布了自动 skill improvement 工具，把昨天的手动 skill lifecycle 变成自动反馈循环。[@PiSquared](https://x.com/PiSquared) [开源了 skill-optimizer](https://x.com/PiSquared/status/2044158570196021596)，用于无需 trial-and-error 地评估、改进和适配 agent skills。[@Saboo_Shubham_](https://x.com/Saboo_Shubham_) [构建了一个 self-improving multi-agent system](https://x.com/Saboo_Shubham_/status/2044110021219553767)，使用 Google Agent Development Kit 和 Gemini 3，让 skills 在 test scenarios 上运行、诊断失败并自动修复。

![使用 Google ADK 和 Gemini 3 自动测试和修复 skills 的 self-improving agent skills 架构](https://pbs.twimg.com/media/HF4jMrpbEAAQpiy.jpg)

[@shannholmberg](https://x.com/shannholmberg) [发布了一张图](https://x.com/shannholmberg/status/2044038207722139688)，展示“self-improving skill loop”：使用 skill、出现问题、修复、告诉 agent “update the skill with the learnings”，然后 skill file 会把 verification steps 和 edge-case warnings 写回自身。口号是：“fix once——every future use of that skill benefits.”

![Self-improving skill loop 流程图：use skill、break、fix、update skill with learnings、permanent improvement](https://pbs.twimg.com/media/HF3h6tnaYAASh_Q.jpg)

[@koylanai](https://x.com/koylanai) [提出](https://x.com/koylanai/status/2043859043446317381)，skills——编码在 markdown procedures 中的知识——才是真正的产品，而 harnesses 是可替换的基础设施。该帖 99 个点赞、98 个收藏，强化了这一点。

**Discussion insight:** [@anatoliygatt](https://x.com/anatoliygatt) 在回复 [@akshay_pachaar](https://x.com/akshay_pachaar) 关于 [MiniMax M2.7 的帖子](https://x.com/akshay_pachaar/status/2044000393110474756)时报告：“30% improvement from harness optimisation alone, zero retraining, weights completely frozen”——这是一个具体数据点，说明 harness/skill optimization 可以替代 model retraining。

**Comparison to prior day:** 昨天，[@avisinghdotdev](https://x.com/avisinghdotdev) 请求 `/update-skills` 命令。今天，三支团队独立发布了自动 skill improvement。从“skills need to evolve”到“here's how they evolve”的缺口在 24 小时内闭合。

### 1.4 Agent Security：“Lethal Trifecta” 🡕

[@iancr](https://x.com/iancr) [回忆](https://x.com/iancr/status/2044050214986633324)，去年 10 月他站在台上警告：“an agentic future where we give agents our logins, credit cards, and identities is a security nightmare.” 六个月后，他说这个预测已经得到验证。在[自回复讨论串](https://x.com/iancr/status/2044050214986633324)中，他定义了“lethal trifecta”：agents 需要访问 value（money、email、credentials、browser、files），但不存在 verification infrastructure 来确保它们安全使用这些权限。该帖获得 41 次 retweets——对安全警告来说传播异常高。

[@omarsar0](https://x.com/omarsar0) [分享](https://x.com/omarsar0/status/2044067923787165799)了 Stanford、KAUST 和 MIT 的 “Multi-User Large Language Model Agents”（arXiv:2604.08567）——第一项系统研究 multi-user LLM agent interaction 的论文。论文发现，在用户目标冲突时，frontier LLMs 无法维持稳定优先级，并且在多轮交互中隐私违规会增加。

![Stanford、KAUST、University of Toronto 和 MIT 的 Multi-User Large Language Model Agents 论文首页](https://pbs.twimg.com/media/HF388ZQasAElVyB.jpg)

[@a_g_e_n_c](https://x.com/a_g_e_n_c) [发布](https://x.com/a_g_e_n_c/status/2044089461567565949)了 devnet task system update，把 marketplace tasks 视为 untrusted input：“A task can describe work, rewards, constraints, and deliverables. But the task text itself has no authority.” task hash 作为 cryptographic work contract。

**Discussion insight:** [@FoltzAI](https://x.com/FoltzAI) 回复 [@rryssf_](https://x.com/rryssf_) 时问“how long until a context curator is the norm?”——把 security 和 context quality 框定为同一个 trust problem 的两面。

**Comparison to prior day:** 昨天记录了 LLM API routers 上的主动攻击（“Your Agent Is Mine”论文）。今天则从攻击记录走向架构响应：“lethal trifecta”框架、multi-user privacy research 和 cryptographic task validation。

### 1.5 Context Noise 被重新定义为核心问题 🡒

[@rryssf_](https://x.com/rryssf_) [发帖](https://x.com/rryssf_/status/2044068259914494459)称：“every major AI agent framework is solving the wrong problem. They're expanding context windows. But the problem isn't capacity——it's that 90% of what agents read is structural noise.” 该帖获得 6 条 quote-tweets 和 66 个收藏，说明引发了实质性讨论。

![展示 structural noise 问题的图：90% agent context 是 noise 而不是 signal](https://pbs.twimg.com/media/HF39P_OasAEHXTw.png)

[@dair_ai](https://x.com/dair_ai) [分享](https://x.com/dair_ai/status/2044066936045351317)了 “Artifacts as Memory Beyond the Agent Boundary”（arXiv:2604.08756），形式化说明环境本身如何充当 agent 的 memory。Artifact Reduction Theorem 证明，某些环境 observations 可以减少 agents 需要内部存储的信息——agents 可以把环境当作 external memory，而不是把所有东西复制进 context。

![Artifacts as Memory Beyond the Agent Boundary 论文封面，引入 Artifact Reduction Theorem](https://pbs.twimg.com/media/HF38CxnbwAA8XYF.jpg)

[@talraviv](https://x.com/talraviv) [观察](https://x.com/talraviv/status/2043828455847145814)：“We solved file sharing 19 years ago, and we still haven't solved shared AI context.” 用 Zapier VP of Product 的话说，目标是“context engineering as a team sport.”

**Comparison to prior day:** 昨天的 context engineering 讨论聚焦 taxonomy（6 components）和 cross-tool memory。今天重新定义了问题：瓶颈不是 context window size，而是 context 中的 signal-to-noise ratio。

## 2. 令人困扰的问题

### Platform Dependency Risk（High）

[@heynavtoor](https://x.com/heynavtoor) [报告](https://x.com/heynavtoor/status/2043940977380073976)，Anthropic 在他 24/7 运行 OpenClaw 时毫无预警地终止了他的 Claude Max subscription（$200/month）：“Anthropic 的一封 email，一切都停了。没有 warning period。没有 migration window。就是结束了。” 截图显示了 Anthropic 的 termination email。

![Anthropic 终止用于连续 coding agent operation 的 Claude Max subscription 的邮件](https://pbs.twimg.com/media/HF2Je2YasAMwwHd.jpg)

这延续了昨天的模式：依赖 subscription 的 workflows 会制造 single points of failure。“always-on agent infrastructure”和“可被撤销的月付 API access”之间的缺口仍是结构性问题。

### Context Noise, Not Context Size（High）

[@rryssf_](https://x.com/rryssf_) [认为](https://x.com/rryssf_/status/2044068259914494459)，agent frameworks 正在通过扩展 context windows 来解决错误问题，而 90% 的内容是 structural noise。多条回复验证了这种挫败感。[@FoltzAI](https://x.com/FoltzAI) 问为什么 frontier labs 还没把 context curation 做成内置功能。框架正在优化 capacity，而从业者需要 signal quality。

### Agent Frameworks 中的 IP Tension（Medium）

[@uniquesingh__](https://x.com/uniquesingh__) [浮现](https://x.com/uniquesingh__/status/2044033790566085082)了一项来自 [@EvoMapAI](https://x.com/EvoMapAI) 的指控：Hermes 团队复制了他们的 “Evolver” agent framework——“same loop, same structure, same reasoning chain.” 该帖获得 81 个点赞和 53 条回复。随着 agent frameworks 激增，围绕架构模式的 IP disputes 正变得更常见。

### Agent Orchestration Skepticism（Medium）

[@alexhillman](https://x.com/alexhillman) [称](https://x.com/alexhillman/status/2044197790692024582)，声称提供“an AI company in a box”的 agent orchestration tools 是“productivity cosplay”。这个批评瞄准 demo-ready agent orchestration 与 production-ready agent management 之间的差距。

### Multi-User Agent Privacy Failures 被记录

Stanford、KAUST、University of Toronto 和 MIT 的 “Multi-User Large Language Model Agents” 论文（arXiv:2604.08567）确认，frontier LLMs 在同时服务多个 users 时，会在多轮交互中出现越来越多 privacy violations，并在 users objectives 冲突时无法维持稳定优先级。这是关于 multi-principal agent problem 的第一项系统研究。

## 7. 机会在哪里

**[+++] Skill Lifecycle Automation。** 今天三支独立团队发布了 skill self-improvement tools（[@PiSquared](https://x.com/PiSquared)、[@Saboo_Shubham_](https://x.com/Saboo_Shubham_)、[@shannholmberg](https://x.com/shannholmberg) 的概念循环）。昨天这还是未满足需求。今天已有三种竞争实现，但没有标准方法。“已有 skills”和“能随时间复利提升质量的 skills”之间的缺口，就是下一个 platform advantage 形成的位置。Databricks 数据显示 78% multi-framework adoption，意味着跨工具工作的 skills 价值更大。（[source](https://x.com/PiSquared/status/2044158570196021596)）

**[+++] Agent Infrastructure for Teams。** Runtime（YC-backed）今天发布，但 team-level agent infrastructure——harnesses、sandboxes、observability、governance——市场仍然大开。[@talraviv](https://x.com/talraviv) 指出 shared AI context 是未解问题。[@ashleyhindle](https://x.com/ashleyhindle) 发布 Fuel。模式一致：个人 agent use 可用；团队 agent use 需要尚未成熟的基础设施。（[source](https://x.com/ycombinator/status/2044113458137051171)）

**[++] Context Curation over Context Expansion。** [@rryssf_](https://x.com/rryssf_) 的框架——90% agent context 是 structural noise——获得 66 个收藏和 6 条 quote-tweets。 “Artifacts as Memory” 论文（arXiv:2604.08756）为能减少内部 context requirements 的 environmental memory 提供理论基础。一个 curate context（去除 noise、优先 signal）而不是 expand context 的工具，将解决已记录的从业者痛点。（[source](https://x.com/rryssf_/status/2044068259914494459)）

**[++] Agent Security Verification。** “lethal trifecta”（[@iancr](https://x.com/iancr)）、multi-user privacy failures（arXiv:2604.08567）和 AgenC 的 cryptographic task validation 都指向同一个缺口：agents 需要 access to value，但不存在 verification infrastructure。Multi-user 论文证明问题不只是理论——frontier models 在 multi-principal conditions 下会主动违反隐私约束。（[source](https://x.com/iancr/status/2044050214986633324)）

**[+] Personal Agent Memory Systems。** GBrain 的 “dream cycle” 概念——夜间 memory consolidation、entity sweeps、knowledge compounding——代表了 agent infrastructure 的新类别。[@KSimback](https://x.com/KSimback) 的研究显示“the moat is context that survives sessions.” Session-scoped agent memory 与持久、复利增长的个人知识之间仍有很大缺口。（[source](https://x.com/garrytan/status/2043948971568312473)）

**[+] Platform-Independent Agent Access。** [@heynavtoor](https://x.com/heynavtoor) 的 Claude Max termination 展示了 subscription-dependent agent workflows 的脆弱性。随着 agents 成为 always-on infrastructure，“revocable subscription”和“mission-critical runtime”之间的不匹配，将催生对 platform-independent agent access layers 的需求。（[source](https://x.com/heynavtoor/status/2043940977380073976)）

## 8. 要点总结

1. **Harness engineering 在一天内从教义走向参考架构。** 比较 10+ tools 的 compatibility matrix、control-system framework（feedforward + feedback）和显式 moat analysis 已经作为社区资源出现。讨论从“what is it?” 转向“how do I implement it?”（[source](https://x.com/codylindley/status/2043911683757035774)）

2. **Skill self-improvement 不再是愿望——三支独立团队发布了实现。** PiSquared 开源 skill-optimizer，Saboo 用 Google ADK 构建 self-improving skills，shannholmberg 则画出了 compound-improvement loop。仅靠 harness optimization（zero retraining）就有 30% performance gain，验证了 skill quality 比 model selection 更重要。（[source](https://x.com/PiSquared/status/2044158570196021596)）

3. **Multi-agent adoption 有了硬数据：20,000 家组织中增长 327%，且 78% 使用两个或更多 frameworks。** Databricks 提供了第一份大规模证据，证明 multi-agent systems 不是实验，而是默认模式。Google 泄露的 Gemini “Agent” platform 证实 major vendors 把 multi-agent workspaces 视为竞争战场。（[source](https://x.com/databricks/status/2044176992120037759)）

4. **Agent security research 正在跑在 agent security infrastructure 前面。** Stanford/MIT multi-user 论文记录了会随多轮交互恶化的 privacy violations。“lethal trifecta” 抓住了结构性问题：agents 需要 value access，却没有 verification。AgenC 的 cryptographic task validation 是今天观察到的唯一 implementation-level response。（[source](https://x.com/iancr/status/2044050214986633324)）

5. **Agentic AI 的护城河正在从 model access 转向 persistent context。** 多个信号汇合：GBrain 用于 knowledge compounding 的 “dream cycle”、KSimback 的 moat analysis 指出 context-that-survives-sessions 是 lock-in，以及 Artifacts-as-Memory 论文将 environmental memory 理论化为 internal context 的替代。Models 正在 commodity 化；session 之间能保留下来的东西才不是。（[source](https://x.com/KSimback/status/2044003658166091968)）

6. **Platform dependency 已成为 production agent workflows 的已记录风险。** Anthropic 无预警、无 migration window 地终止了一个用于 always-on OpenClaw 的 Claude Max subscription。随着 agents 成为 mission-critical infrastructure，可被 subscription revocation 切断的访问形成结构性漏洞，市场尚未解决。（[source](https://x.com/heynavtoor/status/2043940977380073976)）

7. **Context quality，而不是 context quantity，是新兴瓶颈。** “90% agent context 是 structural noise” 的论点在社区中产生共鸣。Artifacts-as-Memory 论文提供了理论替代方案：让 agents 使用环境作为 memory，而不是把所有东西复制进 context windows。Context engineering 正从“how big?” 转向“how clean?”（[source](https://x.com/rryssf_/status/2044068259914494459)）
