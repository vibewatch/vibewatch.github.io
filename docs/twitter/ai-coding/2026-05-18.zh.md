# Twitter AI 编程 - 2026-05-18

## 1. 人们在讨论什么

### 1.1 GitHub Copilot 正在扩展成一个会话平台 🡕

今天最强的 GitHub 讨论串，不是自动补全质量，而是工作流覆盖面。至少 6 条独立帖子把 Copilot 展示成可远程监管的运行时、可编程的上下文层、hooks 的触发目标、终端状态界面，以及自定义自动化和原生应用内部的引擎。这和前一天以计费为中心的讨论相比，是个很有分量的转向：5 月 17 日，Copilot 主要还是按价格和信任来被评判；到 5 月 18 日，信息流花了更多篇幅在讨论以 Copilot 为中心的工作流现在到底能做什么。

[@github](https://x.com/github) [宣布](https://x.com/github/status/2056420570745450864)（93 次点赞，9 条回复，16,681 次浏览）说，GitHub Copilot CLI 和 `@code` 会话的远程控制现已正式可用。关联文档补上了推文里没说清的操作细节：远程控制只适用于交互式会话，需要本地机器保持在线，还可以对组织拥有的席位设置策略控制。[@GHchangelog](https://x.com/GHchangelog) [补充](https://x.com/GHchangelog/status/2056415784356749534)（19 次点赞，1,807 次浏览）还补充说，Copilot Spaces API 也已正式可用，可对 spaces、协作者，以及仓库、文件、issue、pull request 和自由文本等资源做编程式 CRUD。

[@JohnLokerse](https://x.com/JohnLokerse) [分享](https://x.com/JohnLokerse/status/2056236565974696077)（16 次点赞，379 次浏览）分享了一个更具确定性的扩展点：Copilot hooks。他关联的文章描述了一个 `PostToolUse` hook：它会监听 `summary.json`，然后调用 PowerShell、Microsoft MAI-Voice-1 和 Remotion，把结束的会话变成一段带旁白的 changelog 视频。[@jandedobbeleer](https://x.com/jandedobbeleer) [展示](https://x.com/jandedobbeleer/status/2056439557004972148)（42 次点赞，36 次收藏，5,167 次浏览）则展示了一个互补的终端模式：把 Oh My Posh 集成进来，直接在 shell 提示符里显示 Copilot 模型、远程会话状态、会话记录路径和 token 仪表。

![使用 Oh My Posh segment 自定义的 GitHub Copilot CLI 提示符，显示模型、远程状态、token 仪表和会话详情](https://pbs.twimg.com/media/HInu-S-WMAA1qJB.png)

真正落到工作流上的具体构建，出现在这些产品公告之下。[@DThompsonDev](https://x.com/DThompsonDev) [展示](https://x.com/DThompsonDev/status/2056241872356733247)（10 次点赞，4 条回复，1,052 次浏览）展示了一条流程：WhatsApp 语音消息会生成 Jira 工单，并通过 Forge、Gemini 和 Rovo AI 触发由 Copilot 驱动的代码变更；与此同时，[@PerLarsen1975](https://x.com/PerLarsen1975) [指出](https://x.com/PerLarsen1975/status/2056121388616855755)（33 次点赞，3 条回复，3,335 次浏览）表示，只用几个小时、靠 Copilot 加上新的 WinUI agent plugin，就足以做出一个 Windows ARM Winget 管理器。

**讨论要点：** 最强的反对声音，并不是针对远程控制本身，而是针对交接太弱。有人在 GitHub 的发布帖回复说，远程控制只有在交接里包含当前分支、diff、上一条命令、测试状态，以及智能体可触碰范围的明确边界时，才真正有用。

**与前日对比：** 5 月 17 日，Copilot 的主导叙事还是账单震惊和配额混乱，桌面应用只是一个较小的反向信号。到了 5 月 18 日，重心已经转向远程控制、hooks、API、提示符遥测和工作流集成。

### 1.2 Antigravity 仍然是个传闻市场，但证据更具体了 🡕

Google Antigravity 依然是信息流里最响的一条传闻驱动讨论串之一，但这场对话比 5 月 17 日更有证据密度。今天的信息流不再只是笼统地期待 I/O，而是围绕路由行为截图、上手后觉得 “Flash” 确实不一样的反馈，以及对 Google 迟迟没把产品方向讲清楚的直接不满。4 条独立帖子共同支撑了这个主题，而分歧本身也是信号的一部分。

![Antigravity 路由截图，显示 gemini-4-cp 的权重为 1000，而 Gemini 3 为 0](https://pbs.twimg.com/media/HInFRINaYAEVpyb.jpg)

**讨论要点：** Antigravity 的关注度，现在更多由截图取证和带有主观色彩的前后对比体验驱动，而不是官方命名。问题已经不是人们还在不在乎，而是 Google 会不会把这些零散证据整理成一条连贯路线图。

**与前日对比：** 5 月 17 日，Antigravity 还处在 I/O 前的观望期：定价页证明产品仍然存在，但操作层挫败感也很多。到 5 月 18 日，讨论更靠近泄露文化：模型权重、静默替换、版本命名争论。

### 1.3 Codex 的讨论转向运行模式与企业运行时 🡕

Codex 不再主要被当成通用 AI 编程助手来谈，而更像一套需要配置、诊断、持续运行，并部署进企业基础设施的系统。4 条帖子支撑了这个转向：一条讲最佳实践，一条讲 CLI 加固，一条讲常驻执行，还有一条讲混合 / 本地部署 rollout。相比 5 月 17 日，Codex 的讨论明显更偏操作层。

[@godofprompt](https://x.com/godofprompt) [认为](https://x.com/godofprompt/status/2056347845360599177)（40 次点赞，9 条回复，37 次收藏，2,767 次浏览）认为，那些把 Codex 用出最好效果的人，不只是提示词写得更好；他们是在把它当队友用——从结构化提示词一路走到 `AGENTS.md`、技能，最后再到自动化。附带的最佳实践备忘单把这套运行模式压缩成一张图；回复又把这一点说得更尖锐：`AGENTS.md` 里还应该明确写出，智能体必须在哪停手，以及哪些东西绝不能碰。

**与前日对比：** 5 月 17 日，Codex 还更多和配额、重置、迁移对比绑定在一起。到 5 月 18 日，更强的证据已经转向配置、诊断、运行时持久性和企业部署。

### 1.4 大家正把 “Vibe coding” 重新理解为带监督的调试，而不是委托 🡕

今天另一个较小但稳定的主题，是对那种放手式“只要给它提示词就行”话术的反弹。公开案例都指向同一方向：AI 要真能帮你把东西交付出去，前提是人得把范围收紧、审查输出、并调试那些出错的地方。这比 5 月 17 日围绕技能退化的调侃，立场要更明确。

## 2. 令人困扰的问题

### 账号边界与管理员可见性依然不透明

信息流里最尖锐的焦虑讨论串，来自 [@AishwaryaDevv](https://x.com/AishwaryaDevv)，她 [提问](https://x.com/AishwaryaDevv/status/2056208155550593477)（281 次点赞，117 条回复，190,294 次浏览）：如果有人不小心在个人私有仓库上用了公司的 GitHub Copilot 席位，会发生什么？讨论焦点并不是源代码泄露，而是企业管理员到底能看到什么。回复主要围绕活动元数据、仓库可见性和异常 token 消耗激增，而作者也明确说，真正可怕的不是内容暴露，而是“奇怪的用量峰值看起来很可疑”。严重度：高。

同样的可见性问题，也出现在远程控制的讨论里。有人在 [@github](https://x.com/github) 的 [发布帖](https://x.com/github/status/2056420570745450864) 回复说，只有当交接里包含当前分支、diff、上一条命令、测试状态，以及智能体接下来被允许碰什么时，这个功能才值得信任。这些讨论合在一起，指向同一个挫败点：团队现在可以在更多界面里运行智能体了，却依然缺少清楚的回执，去说明智能体做了什么、以及管理员能观察到什么。值得构建：是。这个需求立刻就能落地，边界具体，而且已经在影响真实的职场行为。

### 免费与开放的编程技术栈在运行时依然会出故障

[@badlogicgames](https://x.com/badlogicgames) [警告](https://x.com/badlogicgames/status/2056303762470256881)（182 次点赞，9 条回复，11,467 次浏览）警告说，依赖 Pi 免费模型路径的 OpenCode 用户，要预期会碰到很硬的限流，因为避免限流所需的请求头并不会在那里被加上。他在回复里澄清，这个问题只发生在经由 zen/go 使用免费模型时；付费或非免费模型路径应该没事。对那些想围绕“免费”容量搭真实工作流的重度用户来说，严重度：高。

[@LukeParkerDev](https://x.com/LukeParkerDev) [补充](https://x.com/LukeParkerDev/status/2056211824794853412)（21 次点赞，4,889 次浏览）还补充了另一个与 OpenCode 相邻的基础设施故障：Bun 临时文件带来的磁盘压力。关联的 Bun 问题单里写道，`bun:ffi` 可能会泄漏隐藏的临时原生文件，而正文还明确提到了下游的 OpenCode 报告。当前的应对方式，是手动清理 Bun 临时目录，并放弃“免费模型”或“本地运行时”在运维上也免费这种假设。值得构建：是。这是个典型的可靠性切入口，集中在路由卫生、临时文件清理和后台维护上。

### 模糊委托依然会产出薄弱结果

最干净的可视化失败案例来自 [@robinebers](https://x.com/robinebers)，他 [发帖](https://x.com/robinebers/status/2056373263069241429)（5 次点赞，4 条回复，293 次浏览）展示了自己让 Codex 做一个“类似 Typeform 的获客页”后得到的结果。产出只是一个单薄、明显没做完的原型图，离真正可用还很远。[@PrajwalTomar_](https://x.com/PrajwalTomar_) [概括](https://x.com/PrajwalTomar_/status/2056305557276176894)（281 次浏览）则更直白地概括了更大的模式：新手以为 AI 是魔法，直到数据库迁移坏掉、API 结构变了，或者构建直接失败。[@DavidOndrej1](https://x.com/DavidOndrej1) [认为](https://x.com/DavidOndrej1/status/2056382022700962160)（293 次浏览）主张“要做的是智能体工程，不是 vibe coding”，包括小 PR、显式规划和反馈回路，而不是放手委托。严重度：中到高。

当前的应对方式并不是放弃智能体，而是重新把监督加回来。人们会把任务范围卡得更紧、偏好智能体真正看得见系统的纯代码技术栈，并在信任输出前加上审查回路。值得构建：是，但机会在工作流护栏和调试辅助，而不是再去承诺“智能体自己就能把一切都做完”。

## 3. 人们期望的功能

### 带回执与策略清晰度的远程控制

今天围绕 Copilot 的讨论背后的愿望，不只是“远程访问”。而是带着足够上下文和可审计性、因此值得信任的远程访问。最清晰的表述，来自有人回复 [@github](https://x.com/github) 的 [发布帖](https://x.com/github/status/2056420570745450864)（93 次点赞，16,681 次浏览）：只有当交接里包含当前分支、diff、上一条命令、测试通过 / 失败状态，以及允许范围时，远程控制才真正有用。[@AishwaryaDevv](https://x.com/AishwaryaDevv) [提问](https://x.com/AishwaryaDevv/status/2056208155550593477)（281 次点赞，117 条回复，190,294 次浏览）则从另一边提出了相邻的策略问题：如果在错误的账号上使用 Copilot，企业管理员到底能看到什么？这不是情绪化需求，而是很实际的需求。机会评级：直接。

### 当人离开后仍能继续工作的常驻智能体

[@WesRoth](https://x.com/WesRoth) [分享](https://x.com/WesRoth/status/2056223249965396445)（20 次点赞，6 条回复，1,230 次浏览）分享了 Codex for macOS 泄露出来的 “Locked use” 设置，回复把要解决的任务说得很清楚：人们想合上电脑盖，让智能体继续跑。关联的 GitHub 远程控制文档则从另一侧暴露了同样的需求：要做远程监管，本地机器必须保持在线。人们真正想要的是一个耐久运行时，而不是脆弱的一机一会话。机会评级：直接。

### 拥有正式命名、更清晰 UX 和公开路线图的 Antigravity

[@LexnLin](https://x.com/LexnLin) [提出](https://x.com/LexnLin/status/2056371666393890868)（214 次点赞，9 条回复，7,237 次浏览）在 Google I/O 前用愿望清单的形式把这个需求说了出来：更好的 Antigravity，以及更好的 UX。当天其余证据则解释了原因。[@haider1](https://x.com/haider1) [曝出](https://x.com/haider1/status/2056394697489154174) 一张 `gemini-4-cp` 路由截图，[@HarshithLucky3](https://x.com/HarshithLucky3/status/2056369433237737916) [称](https://x.com/HarshithLucky3/status/2056369433237737916) `3 flash` 这个标签很可能已经在掩盖一个更聪明的模型，而 [@zavxai](https://x.com/zavxai) [提问](https://x.com/zavxai/status/2056020534727745927) Google 为什么干脆不再推动 Antigravity。需求很清楚：用户希望 Google 讲明白，自己现在到底在用什么模型、这款产品是不是战略级的，以及到底有哪些变化正在 rollout。机会评级：理想型。

### 面向非专家的引导式 debug-first 工作流

[@PrajwalTomar_](https://x.com/PrajwalTomar_) [描述](https://x.com/PrajwalTomar_/status/2056305557276176894) 新手 vibe coding 缺的那一块，说到底就是得能扛过迁移、结构变更和构建失败。[@DavidOndrej1](https://x.com/DavidOndrej1) [回应](https://x.com/DavidOndrej1/status/2056382022700962160) 时则给出了一套更结构化的配方：小 PR、显式规划、代码结构技能，以及迫使智能体修复自己产出的反馈回路。这一半是实际需求，一半是教育需求，但它之所以紧迫，是因为用户已经在用这些工具发货了。机会评级：直接且竞争激烈。

## 4. 使用中的工具与方法

| 工具 | 类别 | 评价 | 优势 | 局限 |
|---|---|---|---|---|
| GitHub Copilot / Copilot CLI | AI 编程助手 + CLI | (+/-) | 远程控制已 GA、hooks、Spaces API，可嵌入自定义语音 / Jira 工作流和 WinUI 应用构建 | 管理员可见性仍不清晰，交接回执仍然偏弱，信任依赖更好的可审计性 |
| Antigravity | AI 编程 IDE | (+/-) | 用户需求高、外界感知里的 Flash / Gemini 质量在提升、I/O 前关注度高 | 命名和 rollout 不清晰，证据主要靠传闻驱动，社区不确定这是不是 Google 的战略产品 |
| Codex / Codex CLI | AI 编程智能体 | (+/-) | AGENTS.md 与技能工作流、`codex doctor`、审批模式、本地部署路线、Locked use 进展 | 模糊的设计请求仍然很容易失败，运行时 / 安全边界还在摸索 |
| OpenCode | AI 编程智能体 | (+/-) | 灵活的 zen/go 配置，以及低成本或免费模型路径的吸引力 | 免费模型路由可能遭遇硬限流；用户也得自己承担本地技术栈的运行时怪癖 |
| Bun | JavaScript 运行时 | (-) | 智能体工具链底层使用的本地快速运行时 | `bun:ffi` 临时文件泄漏会在真实的 OpenCode 工作流里造成磁盘压力 |
| Oh My Posh Copilot segment | 提示符 / 状态栏 | (+) | 直接在终端里显示模型、远程状态、会话记录路径、token 仪表和代码行变更 | 需要手动 shell 设置，解决的是可观测性而不是模型质量 |
| Copilot Hooks | 工作流自动化 | (+) | 为工具后置脚本和发布流水线提供确定性的触发点 | 需要自定义脚本，以及 PowerShell、语音模型或渲染工具等外部服务 |
| Copilot Spaces API | 上下文 / 知识 API | (+) | 可对 spaces、协作者，以及仓库、文件、issue 和 PR 等资源做编程式 CRUD | 增加了团队必须治理的策略、范围和访问管理复杂度 |

今天的满意度光谱，一端是把 Copilot 扩展进更深工作流的人，另一端是刚好看到便宜或无人监管配置会在哪里出问题的人。Copilot 的产品覆盖面最宽；Codex 的自主性和运作能力叙事最强；Antigravity 的传闻溢价最高。最显眼的权宜方案，不是整套换工具，而是给工具加上观测层：Oh My Posh 里的提示符遥测、确定性的 hooks、小 PR 审查回路，以及手动清理运行时副作用。最明确的迁移压力，落在那些想要低成本 / 免费访问的 OpenCode 用户身上——他们正在被提醒，路由和本地运行时卫生，和模型质量一样重要。

### Copilot Spaces API 现已正式可用

### Codex CLI 0.131.0 新增面向支持场景的诊断能力

## 7. 机会在哪里

**[+++] 会话治理与交接遥测** —— [@AishwaryaDevv](https://x.com/AishwaryaDevv) 暴露了用户对 Copilot 席位边界和管理员可见性的信心有多低，而有人回复 [@github](https://x.com/github/status/2056420570745450864) 时，则把缺失的交接工件说得很明确：分支、diff、上一条命令、测试和允许范围。[@jandedobbeleer](https://x.com/jandedobbeleer/status/2056439557004972148) 已经在用提示符遥测填补可观测性这一侧，而 Spaces API 又提供了一个可编程管理上下文的位置。这里最强的机会，是做一层把回执、预算可见性、策略和远程会话状态合在一起的治理层。

**[+++] 面向长时运行智能体的耐久自主运行时** —— Codex “Locked use” 泄露、GitHub 远程控制，以及 Dell / OpenAI 的混合部署 rollout，都指向同一个需求：当人离开时，智能体仍能继续工作，但必须处在受控条件下。这个市场信号之所以强，是因为它出现在技术栈的每一层——消费者笔记本工作流、CLI 设置，以及企业基础设施。这里真正的赢家，不是单纯“跑得更久”，而是“在可信、可控的前提下跑得更久”。

**[++] 面向开放与本地智能体技术栈的可靠性工具** —— OpenCode 用户想要低成本访问，但今天的证据也展示了隐藏税：免费模型路径上的 Pi 限流，以及 Bun 临时文件泄漏把磁盘塞满。市场上还有空间容纳这样一层：它能处理路由卫生、感知限流的 fallback、缓存和临时文件清理，以及在会话退化之前做运行时健康检查。这个痛点具体、反复出现，而且不绑定单一供应商。

**[++] 面向 vibe coder 的调试优先脚手架** —— 同一天里 [@robinebers](https://x.com/robinebers/status/2056373263069241429)、[@PrajwalTomar_](https://x.com/PrajwalTomar_/status/2056305557276176894) 和 [@DavidOndrej1](https://x.com/DavidOndrej1/status/2056382022700962160) 的组合，给出了一个清晰的产品切口：引导式工作流，强制任务收敛、审查 diff、检查 migration 和 schema，并把智能体输出送进修复回路。机会中等，因为赛道已经拥挤，但需求真实而且就在眼前。

## 8. 要点总结

1. **GitHub Copilot 的叙事，已从价格争议转向工作流覆盖面。** 5 月 18 日最强的 Copilot 条目，集中在远程控制、hooks、Spaces API、提示符遥测和自定义自动化上，这和 5 月 17 日以计费为主的讨论对比很清楚。([来源](https://x.com/github/status/2056420570745450864))
2. **Antigravity 的关注度依然很高，但 Google 正把沟通工作交给截图取证。** `gemini-4-cp` 路由图和“更聪明的 Flash”报告维持了兴奋感，但官方命名和 rollout 仍不清晰，让整条讨论依旧停留在猜测层。([来源](https://x.com/haider1/status/2056394697489154174))
3. **Codex 正越来越像一个运行环境，而不是聊天界面。** 最佳实践备忘单、`codex doctor`，以及泄露出来的 Locked use 设置，都说明用户群正在摸索如何做到持久运行、诊断和可重复配置。([来源](https://x.com/godofprompt/status/2056347845360599177))
4. **便宜或本地的智能体技术栈，依然埋着基础设施地雷。** OpenCode 用户被提醒，免费模型路径会遭遇 Pi 限流；而关联的 Bun issue 则给出了一个可信根因，解释了隐藏临时文件如何引发磁盘空间问题。([来源](https://x.com/LukeParkerDev/status/2056211824794853412))
5. **现在更成熟的构建者姿态，是“用 AI 提速，但控制权别丢”。** 今天最强的工作流建议，是把任务切小、审查 diff、调试 migration 和 schema，并让反馈回路去修复工作，而不是相信模糊委托。([来源](https://x.com/PrajwalTomar_/status/2056305557276176894))
