# Twitter AI 编程 - 2026-07-03

## 1. 人们在讨论什么

### 1.1 运行框架、交接层和跨模型编排开始变成具体的工作流产品 (🡕)

最技术向的一簇讨论，聚焦的是模型之上的那一层。至少有 4 条高强度内容都在说明，编程智能体如今是按运行框架来被评判的：上下文如何重建、工作如何交接、执行如何验证。相比 7 月 2 日的控制平面讨论，7 月 3 日已经从抽象的编排话题，推进到了具体的协调产品和工作流纪律包。

[@SemiAnalysis_](https://x.com/SemiAnalysis_) [认为](https://x.com/SemiAnalysis_/status/2073149985802105086) (167 点赞，9 条回复，23,691 浏览量，279 收藏数)，编程运行框架之间的差异，与其说来自什么“魔法”，不如说来自它们如何重建上下文、声明工具，以及怎样在规划、行动、验证之间循环。那条讨论串后续的回复把机制讲得更明白：每一轮都会重新打包系统提示词、工具定义和消息历史，而 @Timur_Yessenov 的一条回复又把讨论往前推了一步——真正的产品价值，在于轮次之间哪些内容被装进去、被信任、又被丢掉。

![展示编程运行框架被拆成记忆/上下文、工具/API、规划、执行和反馈循环的示意图](https://pbs.twimg.com/media/HMVO7KOWAAA6ZR7.jpg)

[@theo](https://x.com/theo) [问](https://x.com/theo/status/2072869036615155735) (111 点赞，24 条回复，7,063 浏览量，13 收藏数) T3 Code 是否应该发布一个分支，让 Claude 能拉起 Codex 子智能体，反过来也一样。那条引用推文解释了这为什么重要：他已经在用 Claude 充当架构师，用 Codex 作为特别吃 token 的任务里的可控编码后备方案，而有一条回复直说，多模型编排正是“现在所有团队还在手工拼凑的东西”。

![T3 Code 截图，展示由 Claude 驱动的工作流中拉起 Codex 子智能体](https://pbs.twimg.com/media/HMRPTUJaEAAHfAD.jpg)

[@DanKornas](https://x.com/DanKornas) [把](https://x.com/DanKornas/status/2072875206301851970) (19 点赞，4 条回复，1,385 浏览量，20 收藏数) Engineering Discipline 定位成一层工作流：它通过澄清、规划、执行、审查、调试、清理、优化等明确阶段，把模糊请求变成经过澄清和验证的工作。后续一条 [推文](https://x.com/DanKornas/status/2073147213287719042) (9 点赞，4 条回复，899 浏览量，10 收藏数) 又介绍了 [squad](https://github.com/mco-org/squad)，这是一个拥有 554 个 GitHub 星标的 Rust CLI，用 shell 命令加共享的 SQLite 数据库，在 Claude Code、Gemini CLI、Codex CLI 和 OpenCode 之间协调管理者、执行者和检查者智能体。

**讨论要点：** 最有代表性的回复并不反对智能体。大家要的是更强的路由控制、更明确的交接机制，以及更清楚的规则，说明运行框架该信任什么、该拒绝什么。

**与前日对比：** 7 月 2 日还把运行框架视为真正的产品边界；到了 7 月 3 日，这个判断已经延伸成可安装的协调层和明确的工作流纪律包。

### 1.2 价格、额度和开放权重选项，与模型质量一样左右了工具选择 (🡕)

成本已经和能力本身难以分开。至少有 5 条高强度内容聚焦低成本或开放权重替代方案、套餐比价，或为什么高级智能体方案在财务上越来越说不过去。相比 7 月 2 日聚焦支出可见性，7 月 3 日更像公开比价采购：选对模型、选对价格带、选对备用路径。

[@github](https://x.com/github) [宣布](https://x.com/github/status/2073090339020116476) (341 点赞，33 条回复，41,645 浏览量，63 收藏数)，Kimi K2.7 Code 是 GitHub Copilot 模型选择器里的第一个开放权重模型。回复立刻把这件事从品牌话题变成了工作流问题：用户在问 Copilot 会不会按任务自动推荐合适模型，以及 Kimi 在真实代码仓库工作里能不能扛住 Claude 或 GPT。

[@Alan_Earn](https://x.com/Alan_Earn) [称](https://x.com/Alan_Earn/status/2072923333730214371) (50 点赞，20 条回复，1,201 浏览量，9 收藏数) Kimi K2.7 Code 的成本大约比 Copilot 选择器里的其他选项低 75%。真正有信息量的是那张截图：它显示 Kimi 已经出现在选择器里，标着“开放权重”，而且价格卡片明显低于前沿替代模型。

![GitHub Copilot 模型选择器截图，显示 Kimi K2.7 Code 作为开放权重选项出现，并带有更低的每 token 价格卡片](https://pbs.twimg.com/media/HMSAFQGaAAAo5pF.jpg)

[@kingofknowwhere](https://x.com/kingofknowwhere) [发了一份](https://x.com/kingofknowwhere/status/2072995478850207934) (82 点赞，16 条回复，5,154 浏览量，116 收藏数) 信息密度很高的套餐清单，横跨免费工具、1 美元入门套餐、20 美元类无限方案，以及 100 美元的 Claude Max 或 Codex 5x 订阅。它读起来像一份套利指南，而回复还不断往里补地区定价、仅限邀请的套餐，以及到底哪些方案真能保持无限的讨论。

**讨论要点：** 回复关注的重点，不在排行榜名次，而在路由、利润空间、限流，以及产品到底是把工作流的真实成本暴露出来，还是继续藏起来。

**与前日对比：** 7 月 2 日的重点是支出可见性和静默回退；到 7 月 3 日，这个话题已经扩展成公开比价、开放权重采用，以及明确从托管智能体栈迁出的讨论。

### 1.4 信任与评估开始像独立产品品类 (🡕)

围绕信任的问题，已经从原则讨论转向了具体落地。一个信任评分工具、一篇具体的漏洞利用文章，以及一个偏生产环境的安全调查框架，都在把智能体行为当成需要测试、约束，或包进特定领域工作流里的对象。相比 7 月 2 日的治理讨论，7 月 3 日把“信任”做成了可操作的能力。

[@socialwithaayan](https://x.com/socialwithaayan) [重点提到](https://x.com/socialwithaayan/status/2072952431810654632) (33 点赞，8 条回复，9,366 浏览量，20 收藏数) [iFixAi](https://github.com/ifixai-ai/iFixAi)，他说这个工具会自动发现智能体的真实配置，运行 45 项检查，并在 5 分钟内给出 A-F 评分。公开仓库也印证了这个核心说法，也把它最特别的一点说清了：分数是由另一家厂商的模型打出来的，因为用户不相信智能体会给自己打公正的分。

[@TakSec](https://x.com/TakSec) [概括了](https://x.com/TakSec/status/2073111122996302268) (6 点赞，791 浏览量，3 收藏数) ODIN 的研究：Google Antigravity 可以被藏在 `<title>` 标签里的内容诱导，导出环境变量，并通过渲染出的图片 URL 外传 API key。链接的 [文章](https://www.0din.ai/blog/faking-the-pipeline-data-exfiltration-in-google-antigravity) 把整条链讲得很清楚：只要把提示注入、终端访问和富渲染能力叠在一起，即使用户仍然手动批准步骤，也足以把数据外传出去。

![ODIN 文章截图，描述了通过隐藏标题标签把数据从 Google Antigravity 外传出去的提示注入链](https://pbs.twimg.com/media/HMUrhHvaEAAJpiQ.jpg)

[@tom_doerr](https://x.com/tom_doerr) [分享了](https://x.com/tom_doerr/status/2072786743741341937) (15 点赞，3,557 浏览量，12 收藏数) [Security Investigation Automation System](https://github.com/SCStelz/security-investigator)，这是一个拥有 224 个 GitHub 星标的 Python 项目，把 GitHub Copilot、Microsoft Sentinel、Defender XDR、Graph API、KQL、MCP 服务器和 25 个专门的 Agent Skills 组合在一起。这也是“信任”这个主题的另一面：如果智能体要碰安全运营，开发者就会把它们包进明确的工作流、技能和日志里。

**讨论要点：** 最强的信任信号不是一篇哲学帖，而是用户更愿意用跨厂商评分做评估、用具体漏洞链做安全审查；此外，还有一条声量较小的抱怨——某位 Codex-for-OSS 用户表示，一旦工作沾到安全边界，账号处罚就会显得不透明。

**与前日对比：** 7 月 2 日强调的是治理与审查角色；7 月 3 日补上了测试框架、漏洞文章和可操作的信任工具。

### 托管智能体的经济账和厂商锁定越来越难讲通

### 信任与安全仍然需要额外审计层

## 3. 人们期望的功能

### 内置的跨模型交接

最明确的工作流愿望，是让编排不再靠手工 hack 拼出来。[@theo](https://x.com/theo) [问](https://x.com/theo/status/2072869036615155735) (111 点赞，24 条回复，7,063 浏览量，13 收藏数) T3 Code 是否应该直接内置 Claude-to-Codex 子智能体，而回复里有一句话把更大的需求说得很白：多模型编排正是“现在所有人都在手工拼凑的东西”。[@DanKornas](https://x.com/DanKornas) [推出了](https://x.com/DanKornas/status/2073147213287719042) (9 点赞，4 条回复，899 浏览量，10 收藏数) [squad](https://github.com/mco-org/squad)，恰好就是为了补这个缺口，这也说明需求是现实存在的，不是停留在愿景层面。机会：直接切入。

### 按任务自动路由到模型

用户要的不只是模型选择器里多几个模型，他们更希望产品能自己做聪明的选择。在 GitHub 的 [《Kimi K2.7 Code announcement》](https://x.com/github/status/2073090339020116476) 回复区里，一位用户问 Copilot 会不会按任务自动推荐最佳模型，另一位则说，真正的考验是把工作路由给合适的模型，而不是每次都逼用户手动选择。这是一个由混合模型组合、成本压力和模型能力差异共同驱动的现实需求，而且竞争已经存在，因为不少团队正在自己手搓这套逻辑。机会：竞争型。

### 透明的重置、限额与花费状态

人们希望智能体在切断他们之前，先把还剩多少跑道说清楚。[@hqmank](https://x.com/hqmank) [说](https://x.com/hqmank/status/2072879838814502988) (36 点赞，16 条回复，6,552 浏览量，5 收藏数)，他有 3 次 Codex 重置额度，却不知道它们什么时候开始过期；[@jgonzalezferrer](https://x.com/jgonzalezferrer) [展示了](https://x.com/jgonzalezferrer/status/2072923290977374453) (71 点赞，67 条回复，3,060 浏览量) 一次 Claude 会话限额中断；而 [CodexResets](https://github.com/dfinke/Codex-Resets) 之所以存在，只是因为默认产品视图不够用。这是迫切的现实需求，不是情绪性抱怨。机会：直接切入。

### 保留身份的共享会话与更干净的终端导出

声量较小的诉求反而最具体。[@phraggel](https://x.com/phraggel) [希望](https://x.com/phraggel/status/2073042978591895632) (4 点赞，489 浏览量，2 收藏数) 共享的 OpenCode 会话能保留每个参与者的身份；[@tylerangert](https://x.com/tylerangert) [希望](https://x.com/tylerangert/status/2072771464059601036) (33 点赞，2,154 浏览量，5 收藏数) Codex CLI 的输出复制出来不要有空格问题。今天的信息流里没有任何迹象表明这些是做不到的问题；它们只是产品打磨还没到位。机会：直接切入。

常见的绕行模式是：把高级模型留给架构设计或最终判断，再把执行、研究或通用编码转给更便宜或本地的选择。竞争也在往更上层的堆栈移动：[squad](https://github.com/mco-org/squad)、Engineering Discipline、[iFixAi](https://github.com/ifixai-ai/iFixAi) 和 [CodexResets](https://github.com/dfinke/Codex-Resets) 并不是在打基准测试大战；它们填补的是主流智能体产品仍然留空的协调、验证和额度管理缺口。

## 6. 新动态与亮点

### GitHub Actions 里的 Copilot CLI 不再需要 PAT

[@GHchangelog](https://x.com/GHchangelog) [宣布](https://x.com/GHchangelog/status/2072806286580748318) (35 点赞，2,908 浏览量，10 收藏数)，GitHub Actions 里的 Copilot CLI 现在可以直接使用内置的 `GITHUB_TOKEN`。链接的 [更新日志](https://github.blog/changelog/2026-07-02-copilot-cli-no-longer-needs-a-personal-access-token-in-github-actions/) 把操作层面的影响说得很具体：工作流需要 `copilot-requests: write`，不再需要个人访问令牌，而且由组织拥有的仓库可以把消耗的 AI 额度直接记到组织账上。

### CLI 智能体采用的早期大规模证据终于出现了

[@ComputerPapers](https://x.com/ComputerPapers) [链接了](https://x.com/ComputerPapers/status/2072981233915621518) (81 浏览量，1 收藏数) 一篇 arXiv 论文 [《Adoption and Impact of Command-Line AI Coding Agents》](https://arxiv.org/abs/2607.01418)，研究的是 Microsoft 在 2026 年初推出 Claude Code 和 GitHub Copilot CLI 的情况。这个摘要在这个领域里少见地具体：首次使用主要经由社交网络扩散，留存更取决于编码活跃度，而不是人口统计特征；采用者在 4 个月窗口内合并的 pull request 数，大约比原本会多 24%。

![论文摘要截图，总结了 Microsoft 推出 Claude Code 和 GitHub Copilot CLI 的发现，包括大约 24% 的 pull request 提升](https://pbs.twimg.com/media/HMS1coJXoAEtk8Q.png)

### Anthropic 把智能体技能学习做成了正式课程目录

[@aiwithjonny](https://x.com/aiwithjonny) [强调](https://x.com/aiwithjonny/status/2073039922202919120) (19 点赞，4 条回复，127 浏览量，5 收藏数)，Anthropic 正在提供 18 门免费的官方课程，并附带证书，内容覆盖 Claude Code、MCP、Agent Skills、子智能体、API 工作和通用 AI 素养。值得注意的不只是免费，而是智能体工作流正在被打包成结构化培训，而不再只是散落在各处的讨论串和演示。

## 7. 机会在哪里

**[+++] 支出编排与额度可观测性** —— 证据几乎来自信息流的每一层：横跨几十种套餐的公开价格图、Claude 会话上限打断、Codex 重置过期的困惑，以及用户为了读取重置额度和日期自己做出来的工具。这个机会最强的版本，不是再做一个便宜模型，而是做一层控制层：在会话断掉之前，先说明还剩多少跑道，把工作路由给足够便宜的合格模型，并让重置、限额和计费都可见。([价格调查](https://x.com/kingofknowwhere/status/2072995478850207934), [会话上限](https://x.com/jgonzalezferrer/status/2072923290977374453), [重置过期](https://x.com/hqmank/status/2072879838814502988), [CodexResets](https://github.com/dfinke/Codex-Resets))

**[++] 跨智能体协调与验证层** —— 当天的信息流反复表明，人们在围绕编排缺口造工具，而不是等某个旗舰模型把问题一次性解决。证据涵盖运行框架结构拆解、Claude-to-Codex 子智能体、作为 SQLite 交接层的 `squad`、作为流程层的 Engineering Discipline，以及作为评估层的 iFixAi。([运行框架讨论串](https://x.com/SemiAnalysis_/status/2073149985802105086), [子智能体](https://x.com/theo/status/2072869036615155735), [squad](https://github.com/mco-org/squad), [iFixAi](https://github.com/ifixai-ai/iFixAi))

**[+] 面向智能体 IDE 的安全内容中介层** —— 信任正在成为一个独立的产品层面。Antigravity 漏洞表明，网页读取、终端访问和富渲染能力可以拼成一条外传路径，而 iFixAi 和 Security Investigation Automation System 又说明，开发者已经开始用测试和工作流护栏来应对。这个机会正在浮现，因为需求已经具体，但市场仍然很早也很分散。([ODIN 文章](https://www.0din.ai/blog/faking-the-pipeline-data-exfiltration-in-google-antigravity), [iFixAi](https://github.com/ifixai-ai/iFixAi), [Security Investigation Automation System](https://github.com/SCStelz/security-investigator))

## 8. 要点总结

1. **运行框架设计现在已经是产品本身的一部分，不再是藏在底下的管线。** 最强的技术讨论，讲的是轮次之间如何组装和控制上下文、工具与交接，而不只是底下用的是什么模型。 ([来源](https://x.com/SemiAnalysis_/status/2073149985802105086))
2. **工具选择正在变成跨价格带的组合管理。** Copilot 官方支持开放权重、草根式比价，以及围绕 OpenCode 或本地 GLM-5.2 栈的迁移讨论，都指向同一种行为：高级模型只留给那些真正需要它们的时刻。 ([来源](https://x.com/github/status/2073090339020116476))
