### 1.3 GitHub Copilot CLI走向远程和移动端（🡕）

两篇帖子记录了GitHub Copilot CLI能力的重大扩展。[@msdev](https://x.com/msdev)[演示了](https://x.com/msdev/status/2043769670688461051)使用三个Copilot CLI命令在30分钟内构建的板球计分应用：`/plan`确定范围，`/autopilot`构建，`/fleet`进行扩展。该帖获得77个点赞和47个书签。

[@h34j3j8j](https://x.com/h34j3j8j)[询问了](https://x.com/h34j3j8j/status/2043774154013389201)IntelliJ中Copilot"Ask"模式和"Agent"模式的区别，描述道："用Ask模式，它告诉我在哪里做什么。用Agent模式，它直接替我做。"配图展示了IDE状态栏中的模式下拉菜单（Ask、Edit (Deprecated)、Agent、Plan）。

![GitHub Copilot模式选择器，显示Ask、Edit Deprecated、Agent和Plan选项](https://pbs.twimg.com/media/HFzxUyAXQAAqWdc.png)

## 2. 令人困扰的问题

### Claude Code token膨胀与幽灵token（High）

三个独立来源记录了Claude Code过度消耗token的问题。[@de1lymoon](https://x.com/de1lymoon)基于在Google 11年的经验[报告称](https://x.com/de1lymoon/status/2043668578570744085)："Claude 40%的时间违反代码规范……一个隐藏bug烧掉20,000个幽灵token。"仅一个markdown规则文件就将规范违反率从40%降至3%。[@grok](https://x.com/grok)[证实](https://x.com/grok/status/2043672440966967661)"较新版本的Claude Code（2.1.100+）每次请求会注入约20k额外的服务端token，已通过HTTP代理测试、Reddit分析和GitHub公开issue得到验证。"[@a_lamparelli](https://x.com/a_lamparelli)[展示了截图](https://x.com/a_lamparelli/status/2043710164964983197)，显示单个"continue"命令在没有任何工具调用的情况下消耗了2%的会话配额。`/context`详细分析揭示了开销来源：系统提示词7k token（3.5%）、系统工具8.9k（4.5%）、自动压缩缓冲区33k（16.5%）——这还是在任何用户消息之前。

![Claude Code上下文使用情况，显示200k token中已使用49.5k，按类别详细分类](https://pbs.twimg.com/media/HFy2jBzWkAANn3I.jpg)

这延续了昨日报告中关于缓存TTL和token成本的关切，今日证据增加了具体的版本号和独立验证方法。

## 3. 人们期望的功能

### Token成本透明度与可观测性

[@a_lamparelli](https://x.com/a_lamparelli)的[截图](https://x.com/a_lamparelli/status/2043710164964983197)显示Claude Code的`/context`命令提供了token使用明细，但单个"continue"指令消耗2%会话配额的原因对用户而言仍无从解释。[@grok](https://x.com/grok)[指出](https://x.com/grok/status/2043672440966967661)约20k服务端token的注入并未被记录在文档中。结合昨日报告中的缓存TTL回退问题，用户持续呼吁实时成本可见性和可预测的计费。相关信息在服务端是存在的，但并未以可操作的形式呈现给开发者。

### 感知代码规范的AI编程智能体

[@de1lymoon](https://x.com/de1lymoon)[证明](https://x.com/de1lymoon/status/2043668578570744085)仅一个markdown文件就将Claude Code的规范违反率从40%降至3%。"开箱即用"与"按团队规范正确工作"之间的鸿沟，目前靠手动配置来弥合。一个能自动检测并适应项目规范——lint规则、命名模式、架构决策——而无需手工编写规则文件的智能体，将能解决Claude Code最常见的失败模式。

### 移动优先的智能体化开发

GitHub的[远程CLI会话](https://github.blog/changelog/2026-04-13-remote-control-cli-sessions-on-web-and-mobile-in-public-preview)开始满足这一需求，但该功能尚处于公开预览阶段，移动端可用性有限（Android Beta、iOS TestFlight）。[@msdev](https://x.com/msdev)获得47个书签的帖子以及远程会话公告所释放的需求信号表明，开发者希望能从任何设备监控、引导和审批智能体的工作——而不仅限于运行CLI的那台机器。

## 6. 新动态与亮点

**GitHub Copilot CLI远程会话进入公开预览。** `copilot --remote`命令将CLI活动实时流式传输至GitHub，支持从Web或移动端进行监控和引导。功能包括会话中消息发送、计划审查、模式切换，以及从任何设备审批权限请求。已在Android Beta和iOS TestFlight上可用。（[Changelog](https://github.blog/changelog/2026-04-13-remote-control-cli-sessions-on-web-and-mobile-in-public-preview)）

## 7. 机会在哪里

**[+++] 强信号：自托管智能体基础设施**

Multica在数天内获得4,000+星标，量化了市场对云端锁定型托管智能体平台的自托管替代方案的需求。Anthropic的Managed Agents需要使用其云端和定价体系；企业团队需要可运行在自有服务器上且支持模型选择（Claude、GPT、开源模型）的智能体编排方案。Multica已验证了这一模式——守护进程、工单分配、隔离工作区、实时更新、可复用技能。机会在于将其强化至企业级：SOC 2合规、审计日志、基于角色的访问控制，以及与现有CI/CD管道的集成。任何构建智能体基础设施的团队都应密切关注这一领域。

**[++] 中等信号：AI编程工具的Token成本可观测性**

今日数据为昨日的缓存TTL关切增添了具体证据。[@a_lamparelli](https://x.com/a_lamparelli)的`/context`截图显示，系统开销（提示词 + 工具 + 自动压缩缓冲区）在用户发送第一条消息之前就消耗了约50k token。[@grok](https://x.com/grok)指出v2.1.100+每次请求额外注入约20k服务端token。[@de1lymoon](https://x.com/de1lymoon)报告每个隐藏bug消耗20,000个幽灵token。一个能让这些成本可见、可预测、可优化的代理或仪表板，将服务于每一个触及配额上限的Claude Code用户。数据是存在的，但未以可操作的方式呈现。

**[++] 中等信号：MCP作为通用中间件层**

MCP今日出现在三个不相关的场景中：企业低代码（Power Apps）、Web3游戏（Dungeon Roll）和智能体基础设施（Multica）。[@oscar_ster3808](https://x.com/oscar_ster3808)精准概括了发展轨迹："MCP正在悄悄吞噬中间件栈。"机会在于MCP工具链——MCP服务器的发现、测试、安全扫描和监控——因为该协议正在成为AI智能体与外部系统之间的默认集成层。Anthropic MCP Registry是正在形成的应用商店；帮助开发者发布、验证和维护MCP服务器的工具填补了基础设施空白。

**[+] 新兴信号：非编程人员AI熟练度工具**

[@clairevo](https://x.com/clairevo)对Cowork的定位——"在无需打开终端的情况下教授Claude Code的思维模型"——以及[@degensing](https://x.com/degensing)对9个非开发者工程岗位AI工具需求的梳理，都指向一个日益壮大的受众群体：他们需要AI熟练度，但不需要CLI技能。"Brain"文件模式（一个持久化上下文文档，告诉AI你是谁以及你的工作方式）是一项工作流创新，可以从Cowork推广到任何AI工具。针对特定角色的AI熟练度模板和培训材料（QA、DevOps、SRE、数据）代表着一个未被充分服务的市场。

**[+] 新兴信号：设备无关的智能体化开发**

GitHub的远程CLI会话和Codex的桌面超级应用都在解决同一个约束：智能体化编程工作流被锁定在单一设备上。机会在于支持设备间无缝切换的工具层——在工作站启动任务、从手机监控、在平板上审批权限。GitHub凭借`copilot --remote`拥有先发优势，但这一模式可推广到任何CLI智能体。面向智能体引导（不仅仅是监控）的移动原生界面是下一步方向。
