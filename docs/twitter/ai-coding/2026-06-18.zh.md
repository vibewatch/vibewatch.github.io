# Twitter AI 编程 - 2026-06-18

## 1. 人们在讨论什么

### 1.1 Gemini CLI 的关闭，变成了一场生态信任测试 🡕

6 月 18 日，是 Google 终端策略不再停留在假设层面的一天。共有 3 条保留条目支撑这一主题：Jack Wotherspoon 的维护者告别帖、官方 @geminicli 的过渡通知，以及一条引用 Google 文档并配图的帖子。随后，回复区也从礼貌致谢迅速转向对安装失败、遥测门槛，以及 Antigravity CLI 无法做到 1:1 对等替代的愤怒。

[@JackWoth98](https://x.com/JackWoth98) [告别](https://x.com/JackWoth98/status/2067697673671450742) Gemini CLI 用户（262 个点赞、44 条回复、31,344 次浏览、30 次收藏），并明确表示 Google AI Pro、Ultra 和免费层个人账户已在 6 月 18 日停止可用。语气本身很关键：他称这一天是“令人难过的一天”，并在回复里说这个项目并没有按他希望的方式结束，这让这次迁移更像被强加，而不是值得庆祝的升级。

[@geminicli](https://x.com/geminicli) [确认](https://x.com/geminicli/status/2067702889837953512) 了同样的截止点（72 个点赞、18 条回复、6,146 次浏览、13 次收藏），并在官方渠道里划清了个人用户和企业用户的边界：Code Assist license 持有者和 API key 用户不受影响。Google 自己的公开博客则表示，这次调整是为了收拢到 Antigravity 统一的多智能体后端，同时也承认，在发布时它并没有做到 1:1 的功能对等 ([blog](https://developers.googleblog.com/an-important-update-transitioning-gemini-cli-to-antigravity-cli/))。

[@ai_for_success](https://x.com/ai_for_success) [放大传播](https://x.com/ai_for_success/status/2067484067269063084) 了最具体的证据（55 个点赞、9 条回复、4,102 次浏览、7 次收藏）：一张 Google 迁移通知截图，明确写着 Gemini CLI 和 Gemini Code Assist IDE extensions 将在 6 月 18 日停止服务于个人 Pro、Ultra 和免费用户。

![Google Developers 的迁移通知，显示 Gemini CLI 和 Gemini Code Assist 将于 2026 年 6 月 18 日停止服务 Google AI Pro、Ultra 和免费个人用户](https://pbs.twimg.com/media/HLEtzWraAAEEviS.jpg)

[@robinebers](https://x.com/robinebers) [报告](https://x.com/robinebers/status/2067533536706613319)（25 个点赞、10 条回复、1,664 次浏览），Antigravity IDE 如果不开启 telemetry 就无法继续；而 [@_Creation22](https://x.com/_Creation22) [展示](https://x.com/_Creation22/status/2067294279954043375) 了一次卡在会话中途的安装流程（17 个点赞、4 条回复、325 次浏览）。这些抱怨把这次迁移变成了一个信任问题，而不只是改名。

**讨论要点：** 最强烈的回复并不是在怀旧，而是在追问现实里到底坏了什么：安装循环、Antigravity 和 Antigravity IDE 之间让人困惑的产品拆分，以及有人直白地说，Google 在强推迁移之后，“把新模型、CLI 和 Antigravity 都搞砸了”。

**与前日对比：** 相比 6 月 17 日，当时 Antigravity 还只是 Google 更大编程产品栈讨论中的一个界面；到了 6 月 18 日，这次转向已经变得不可逆，而且带有明确截止日期。

### 1.2 工作流捕获与操作层，进一步走到中心位置 🡕

第二个主要主题不是某个新模型发布，而是更好的方式去把围绕模型的工作打包起来。共有 5 条保留条目支撑这一主题：Codex Record & Replay、那条关于 CLAUDE.md 错误率的讨论串、AI DevKit、gstack 和 ClaudeKit Skills。

[@mark_k](https://x.com/mark_k) [发布](https://x.com/mark_k/status/2067689265941885253) 了 6 月 18 日的 Codex app changelog（205 个点赞、19 条回复、13,061 次浏览、28 次收藏），其中最醒目的更新是 Record & Replay：先在 macOS 上给 Codex 演示一次工作流，再把它变成一个可复用 skill，之后还能继续管理。OpenAI 的 changelog 也补充了塑造反馈的重要限制条件：它依赖 Computer Use，而且在 EEA、英国和瑞士上线时不可用 ([changelog](https://developers.openai.com/codex/changelog))。

![Codex app 26.616 的更新日志截图，展示了 Record and Replay、批量自动化历史操作、SSH deep links 和 Browser Use 改进](https://pbs.twimg.com/media/HLHn16EWIAAAE3C.jpg)

[@DevAdventur3s](https://x.com/DevAdventur3s) [展示](https://x.com/DevAdventur3s/status/2067689725100458003) 了同一功能面向用户的一侧（34 个点赞、1 条回复、2,874 次浏览、10 次收藏）：一个插件面板可以记录点击、输入文本和窗口内容，最长 30 分钟，然后把这次运行变成一个可通过提示词调用的 skill。这件事之所以重要，是因为它把“自动化”从营销语言变成了一个可检查的 UI。

![Codex Record and Replay 插件面板，展示了示例提示词，以及如何把录制的 macOS 工作流变成可复用 skill 的说明](https://pbs.twimg.com/media/HLHon82WoAAMqIF.jpg)

[@cyrilXBT](https://x.com/cyrilXBT) [认为](https://x.com/cyrilXBT/status/2067419526019387560)（60 个点赞、5 条回复、4,526 次浏览、11 次收藏），大多数编程智能体失败的原因，不是模型能力太弱，而是上下文问题；他把 41% 到 11% 再到 3% 的错误率下降，总结为 CLAUDE.md 指引越明确，效果越好。这个观点不仅来自评论者，也得到了构建者呼应：[@DanKornas](https://x.com/DanKornas) [发布](https://x.com/DanKornas/status/2067593408697311734) 了 AI DevKit（10 个点赞、6 条回复、559 次浏览、8 次收藏），把它定位为带验证闸门和本地记忆的工作流控制平面；[@oliviscusAI](https://x.com/oliviscusAI) [提到](https://x.com/oliviscusAI/status/2067660787653410989) Garry Tan 新近开源的 gstack 工作流（5 个点赞、1 条回复、184 次浏览、4 次收藏）；而 [@tom_doerr](https://x.com/tom_doerr) [链接](https://x.com/tom_doerr/status/2067492857028489266) 了 ClaudeKit Skills（8 个点赞、426 次浏览、6 次收藏），把它做成一个可在 marketplace 安装的可复用智能体技能包。

**讨论要点：** 回复区同时朝两个方向发力：人们喜欢从一次性提示词走向可重复工作流的转变，但只要这些工作流被锁死在某个 OS、某个市场，或某一个智能体界面里，他们也会立刻抱怨。

**与前日对比：** 6 月 17 日强调的是产品之间的桥接；6 月 18 日则更进一步，强调产品内部可复用的操作层。

### 1.3 GitHub 和 VS Code 把模型控制做成了更显性的产品界面 🡕

第三个讨论簇显示，GitHub 和 VS Code 正在把模型选择、路由和自动化，从隐藏默认值做成看得见的产品界面。共有 4 条保留条目支撑这一主题。

[@pierceboggan](https://x.com/pierceboggan) [解释](https://x.com/pierceboggan/status/2067449111809597803) 了 Copilot Auto 现在如何使用路由模型（29 个点赞、4 条回复、1,231 次浏览、5 次收藏）：它会综合推理深度、代码复杂度、调试难度，以及工具编排需求。GitHub 配套文章则把机制写得很清楚：通过 prompt caching、延迟工具加载和 HyDRA router，在任务需要时继续向上路由，同时减少 token 浪费 ([blog](https://github.blog/ai-and-ml/github-copilot/getting-more-from-each-token-how-copilot-improves-context-handling-and-model-routing/))。

![HyDRA 基准测试图，展示 GitHub Copilot Auto 如何在五模型路由池中权衡解决质量与成本节省](https://pbs.twimg.com/media/HLENrQwWUAAFe5t.jpg)

[@pierceboggan](https://x.com/pierceboggan) [还强调](https://x.com/pierceboggan/status/2067638151997452597) 了一点（23 个点赞、3 条回复、1,211 次浏览、5 次收藏）：VS Code Chat 现在无需任何 GitHub 账户或 Copilot 订阅，也能使用 BYOK 提供商。Microsoft 自己的说明明确了范围：BYOK 可用于 chat 和 agent 工作流，包括离线本地模型配置，但不覆盖内联补全或依赖 embedding 的功能 ([blog](https://code.visualstudio.com/blogs/2026/06/18/byok-vscode))。

[@github](https://x.com/github) [宣布](https://x.com/github/status/2067704358821597555) MAI-Code-1-Flash 已在更多 Copilot 界面中可用（80 个点赞、4 条回复、10,455 次浏览、15 次收藏）；[@_Evan_Boyle](https://x.com/_Evan_Boyle) [补充](https://x.com/_Evan_Boyle/status/2067378714476953926) 说，Copilot App 在个人 Pro、Pro+ 和 Max 计划上的注册也重新开放了（40 个点赞、4 条回复、2,087 次浏览、6 次收藏）。这几条帖子放在一起，让 6 月 18 日看起来像是 GitHub 对 6 月 17 日 GA 发布后的跟进日。

**讨论要点：** 主要的反对声并不针对路由本身，而是针对控制权。回复区想知道 Auto 什么时候会选更便宜的模型、MAI 是否会进入路由池，以及用户究竟能多大程度直接操控这条工作流。

**与前日对比：** 6 月 17 日把 Copilot App 定义成操作界面；6 月 18 日则补上了它底下的控制机制：Auto 路由、BYOK，以及更广泛的 MAI 模型可用性。

### 1.4 以智能体为原生核心的设计与游戏工作流继续扩张，但回复区依旧怀疑 🡒

一个更小但持续存在的主题是，AI 编程正在渗入设计和游戏制作界面。共有 3 条保留条目支撑这一主题。

[@TheShortBear](https://x.com/TheShortBear) [把](https://x.com/TheShortBear/status/2067410669800956237) Unreal Engine 5.8 的实验性 MCP server 支持，描述为迈向用户自定义“高级 Roblox”式游戏创作的一步（58 个点赞、11 条回复、15,044 次浏览、15 次收藏）。被引用的 Unreal 帖子之所以重要，是因为它明确写道：团队可以把任何智能体接入引擎源码、流水线和工作流中。

[@OpenDesignHQ](https://x.com/OpenDesignHQ) [把](https://x.com/OpenDesignHQ/status/2067551675963822444) Open Design 定位成 Claude Design 的开源替代品（6 个点赞、2 条回复、447 次浏览、7 次收藏），重点在画布编辑、插件和 22+ 个智能体。公开仓库描述又把这个思路推进了一步：它是一个本地优先的设计工作空间，带 150 个设计系统、261 个插件、BYOK 模型路由，并支持 Claude Code、Codex、Copilot、Antigravity 等编程智能体 ([repo](https://github.com/nexu-io/open-design))。

[@kr0der](https://x.com/kr0der) [指出](https://x.com/kr0der/status/2067737123986538719) 了 Claude Design 的 `/design-sync` 流程（2 个点赞、1 条回复、62 次浏览）：它能读取代码库的设计系统，并复用现有组件。这张截图虽然互动量低，但信号很强，因为它展示了设计工具边界正在如何移动。

**讨论要点：** 回复区远没有发布文案那样乌托邦。在 Unreal 那条讨论串下，人们立刻担心 AI slop、审核、持久性，以及这个实验性插件在 Linux 上到底能不能正常工作。

**与前日对比：** 前一天更多是在讨论编程工具之间的工作流可迁移性。6 月 18 日则把同样的逻辑延伸到设计和游戏制作里，但身边的怀疑也更强。

## 2. 令人困扰的问题

### 被强推的迁移与频繁出错的配置流程

严重程度：高。当天最大的挫败感，不是模型质量，而是在一个新界面尚未准备好之前就被强制切过去。[@JackWoth98](https://x.com/JackWoth98) 的帖子 [让](https://x.com/JackWoth98/status/2067697673671450742) 个人 Gemini CLI 用户的截止时间变得毫无歧义（262 个点赞、44 条回复、31,344 次浏览、30 次收藏）；与此同时，Google 自己的迁移文章也写明，Antigravity CLI 目前还做不到与 Gemini CLI 的 1:1 功能对等 ([blog](https://developers.googleblog.com/an-important-update-transitioning-gemini-cli-to-antigravity-cli/))。站在用户这一侧，[@robinebers](https://x.com/robinebers/status/2067533536706613319) [表示](https://x.com/robinebers/status/2067533536706613319)（25 个点赞、10 条回复、1,664 次浏览），Antigravity IDE 如果不开 telemetry 就打不开；而 [@_Creation22](https://x.com/_Creation22) [展示](https://x.com/_Creation22/status/2067294279954043375) 了一条一直转圈的安装流程（17 个点赞、4 条回复、325 次浏览），回复区则说现在可能需要同时安装 Antigravity 和 Antigravity IDE。人们当前的应对方式，是能拖就拖、退回到仍受支持的 enterprise/API-key 路径，或者把多个 Google 界面反复重装，直到其中一个能用。这值得做成产品，因为失败发生在任何真正有产出的编码开始之前。

![Antigravity 安装截图，显示应用在配置过程中卡住；回复区后来又补充说，现在可能需要同时安装 Antigravity 和 Antigravity IDE](https://pbs.twimg.com/media/HLCBFgcbIAArcLH.jpg)

### 仓库上下文仍然显得不够，而且不透明

严重程度：高。[@sudoingX](https://x.com/sudoingX) [发问](https://x.com/sudoingX/status/2067701388709364130)（16 个点赞、5 条回复、1,474 次浏览），为什么 Codex 仍然不默认给订阅用户 1M 上下文模式；他把痛点压缩成一句话：用户不该去调试智能体到底有没有读够整个仓库。[@cyrilXBT](https://x.com/cyrilXBT) [则用了](https://x.com/cyrilXBT/status/2067419526019387560) 相反的表述，得到同样结论（60 个点赞、5 条回复、4,526 次浏览、11 次收藏）：多数失败来自上下文缺失，而不是模型太弱，并引用了从 41% 到 11% 再到 3% 的错误率下降，说明随着指引变得更明确，问题会明显收敛。构建者现在的应对方式，是加各种 sidecar，而不再信默认值：AI DevKit 增加本地记忆和验证闸门，gstack 和 ClaudeKit 则在智能体外面再包一层更明确的工作流规则。这值得做成产品，因为这种痛点是结构性的，而且会在不同工具里反复出现，而不只是某个厂商的一条讨论串。

### 新能力仍然被硬门槛挡在后面

严重程度：中。当天还反复出现了另一种挫败感：一些原本很吸引人的功能，总是伴随着访问限制。[@mark_k](https://x.com/mark_k) [强调](https://x.com/mark_k/status/2067689265941885253) 了 Codex Record & Replay（205 个点赞、19 条回复、13,061 次浏览、28 次收藏），但回复区马上抱怨它只支持 macOS；而 OpenAI 自己的 changelog 也写明，它依赖 Computer Use，并且在 EEA、英国和瑞士首发不可用 ([changelog](https://developers.openai.com/codex/changelog))。在 GitHub 这边，[@pierceboggan](https://x.com/pierceboggan/status/2067665332307943621) [表示](https://x.com/pierceboggan/status/2067665332307943621) MAI-Code-1-Flash 正在向个人 Copilot 计划推出（48 个点赞、7 条回复、1,419 次浏览），但回复区仍然在问模型到底现在能不能看见，以及混合模型行为是否可调试。当前的绕行方式，是等灰度发布、换别的地区或设备，或者继续用能力更弱但更可预测的旧流程。这值得做成产品，但切口没那么大，因为平台厂商已经在陆续交付部分修复方案。

## 3. 人们期望的功能

### 持久的仓库记忆与上下文控制

最清晰的实际愿望，是让智能体能够稳定理解仓库，而不是让用户去猜。[@sudoingX](https://x.com/sudoingX) [把](https://x.com/sudoingX/status/2067701388709364130) 这个诉求压缩成一个上下文问题（16 个点赞、5 条回复、1,474 次浏览）：在一个智能体式编程产品里，用户不应该去调试工具到底有没有读够整个代码库。[@cyrilXBT](https://x.com/cyrilXBT/status/2067419526019387560) [则给出](https://x.com/cyrilXBT/status/2067419526019387560) 了工作流答案，而不是模型答案（60 个点赞、5 条回复、4,526 次浏览、11 次收藏）：真正能把错误率压下来的，是明确规则和检查点。AI DevKit、gstack 和 ClaudeKit 之所以存在，就是因为这种需求既现实又紧迫：人们想让记忆、约定和验证机制跨会话保留下来，而不必把每一次提示词都堆得更臃肿。机会：直接。

### 对 secret 安全的智能体执行

第二个需求，是让智能体能碰真实系统，却不用把 secret 拉进聊天记录。[@glcst](https://x.com/glcst) [发布](https://x.com/glcst/status/2067576762481938545) 了 keymaxxer（19 个点赞、4 条回复、857 次浏览、7 次收藏），专门把 token 放在本地加密 vault 中，让智能体按名称请求 secret，并在敏感执行时要求批准。它的仓库描述把潜台词写得非常明白：现在更常见的替代方案，是把 API key 粘进 shell、提示词，或者两边都粘。这是一个具有直接运营价值的实际需求，因为智能体时代会让更多命令、工具和提供商触碰到凭据。机会：直接。

### 能跨厂商和界面变化存活下来的工作流包

信息流还显示，人们希望工作流能跟着走，即便前端工具变了。Google 强推个人 Gemini CLI 用户迁移到 Antigravity；GitHub 推出 BYOK 和 Auto；构建者则用 AI DevKit、gstack、ClaudeKit Skills 和 Open Design 这类可迁移层来回应。[@DanKornas](https://x.com/DanKornas/status/2067593408697311734) [把](https://x.com/DanKornas/status/2067593408697311734) AI DevKit 描述成一套可在 Claude Code、Cursor、Codex CLI、Gemini CLI 和 Copilot 之间重复使用的工程工作流；[@tom_doerr](https://x.com/tom_doerr/status/2067492857028489266) [链接](https://x.com/tom_doerr/status/2067492857028489266) 了一个可复用 skill marketplace；[@OpenDesignHQ](https://x.com/OpenDesignHQ/status/2067551675963822444) [则把](https://x.com/OpenDesignHQ/status/2067551675963822444) 同样的可迁移性推进到了设计产物里。这个需求很实际，但竞争也越来越激烈，因为已经有多个构建者从不同方向攻击同一层可迁移性问题。机会：竞争型。

## 4. 使用中的工具与方法

| 工具 | 类别 | 评价 | 优势 | 局限 |
|---|---|---|---|---|
| Antigravity CLI / IDE | 编程智能体界面 | (+/-) | 统一多智能体后端、基于 Go 的 CLI 更快、保留了 Gemini CLI 的关键能力、企业路径仍受支持 | 对个人用户是强制迁移、发布时没有 1:1 对等能力，而且存在安装与 telemetry 抱怨 |
| OpenAI Codex app | 智能体工作空间 | (+/-) | Record & Replay 把演示过的 macOS 工作流变成可复用 skills；Browser Use 和自动化控制也在持续扩展 | 新功能只支持 macOS，且在 EEA/英国/瑞士不可用，用户仍在质疑上下文上限 |
| GitHub Copilot Auto + App | 智能体工作空间 / 路由层 | (+) | 能按任务感知做模型路由、支持 prompt caching、延迟工具加载、重新开放个人访问，以及可见的自动化工作流 | 用户仍希望路由过程更透明、模型控制更明确，灰度发布滞后更少 |
| VS Code BYOK | 编辑器模型层 | (+) | 不需要 GitHub 账户或 Copilot 计划、支持提供商 key 和本地模型、覆盖 chat 与 agent 工作流 | 不覆盖内联补全或强依赖 embedding 的功能；提供商计费仍然分开 |
| AI DevKit | 智能体控制平面 | (+) | 一个配置、一个控制台、本地记忆、跨智能体消息、验证闸门，以及跨工具共享生命周期 | 需要额外安装和维护一层；价值取决于团队执行纪律 |
| keymaxxer | Secret 管理器 / MCP | (+) | 让 secret 不进入对话上下文，只在命令执行时注入，并为敏感用途增加审批 | 仍需要本地解锁/批准流程，也不能消除同一用户机器上的风险 |
| Open Design | 智能体式设计工作空间 | (+/-) | 本地优先、开源、设计系统加插件、可接多种编程智能体 CLI 和 BYOK 模型路由 | 独立设计界面本身仍在稳定中，而且生态广度也增加了配置复杂度 |
| gstack | Claude Code 工作流工具包 | (+) | 把 Claude Code 变成一个带规划、审查、QA、安全和发布步骤的结构化工程团队系统 | 工作流立场很强，而且中心仍是 Claude Code，而非完全与模型无关 |
| ClaudeKit Skills | skill 市场 / 工作流库 | (+) | 可安装 skill 包、覆盖 context engineering，并带 MCP-management subagent、可复用仓库工作流 | 主要对已经投入 Claude Code 插件模式的团队更有价值 |

整体满意度更偏务实而不是意识形态化。人们会奖励那些把上下文、自动化或安全边界讲清楚的工具；一旦同样的产品把限制藏在灰度发布、OS 限制或迁移断崖后面，他们也会立刻反感。

最清晰的迁移模式，是摆脱对单一界面的依赖。Google 把个人用户从 Gemini CLI 推向 Antigravity，而 GitHub 和 VS Code 则朝相反方向走：通过 Auto、BYOK 和 MAI-Code-1-Flash 扩大模型选择。

竞争分野已经不再只是模型对模型。它变成了厂商原生工作空间对可迁移控制平面、原始上下文窗口对记忆/检索层，以及直接 shell 访问对带审批的 secret 执行。

## 7. 机会在哪里

**[+++] 不怕迁移的智能体操作层** —— 证据同时来自第 1、3、4、5 节：Google 把用户从 Gemini CLI 上强行迁走，GitHub 和 VS Code 扩大模型选择，而构建者则用 AI DevKit、gstack、ClaudeKit Skills 和 Open Design 做出回应。最强的切入口，是一层即便底层智能体界面变化，也能保住记忆、规则和交接的工作流层。

**[+++] 面向智能体上下文、验证与 secret 的信任基础设施** —— 第 2 节关于上下文的抱怨、第 3 节对持久仓库记忆和 secret 安全执行的需求，以及第 5 节里 AI DevKit 和 keymaxxer 的发布，都指向同一个空缺。这个机会很强，因为痛点是运营层面的，而且非常即时：用户并不信智能体看得够多、验证得够充分，或足够安全地处理了凭据。

**[++] 可见的路由与 BYOK 控制平面** —— GitHub 的 HyDRA/Auto 工作、VS Code 的 BYOK 推出，以及 MAI-Code-1-Flash 的扩展，都说明用户希望模型选择是明确且可操控的。这个机会属于中等强度，因为主要厂商已经在这里发力，但回复区仍显示，用户对成本滑杆、更好的调试可见性以及更顺滑的铺开方式仍有未被满足的需求。

**[+] 以智能体为原生核心的设计与游戏流水线** —— Unreal 的 MCP 支持、Open Design，以及 Claude Design 的 `/design-sync` 流程，都指向一个更大的智能体栈，覆盖代码、设计和内容。这个信号仍在浮现，而非主导趋势，因为同样的讨论串里也不断冒出对 AI slop、审核以及集成到底是否稳定可用的担心。

## 8. 要点总结

1. **6 月 18 日把 Google 的编程智能体转向，变成了一次被强制执行的迁移事件，而不是渐进式偏好变化。** 维护者告别、官方 Gemini CLI 通知和公开的 Google 迁移文章都在传达同一件事：不管个人用户是否准备好，这一天都被切换过去了。([source](https://x.com/JackWoth98/status/2067697673671450742))
2. **工作流捕获和流程封装，比起新的模型发布更重要。** Codex Record & Replay、CLAUDE.md 纪律、AI DevKit、gstack 和 ClaudeKit Skills，都指向围绕模型构建可复用操作层，而不是只对原始权重本身兴奋。([source](https://x.com/mark_k/status/2067689265941885253))
3. **GitHub 和 VS Code 的竞争点，已经是路由、模型选择和智能体运营，而不只是自动补全。** HyDRA 路由、无需 Copilot 账户的 BYOK、重新开放的 Copilot App 注册，以及 MAI-Code-1-Flash 的可用性，都扩大了围绕编程工作的控制界面。([source](https://x.com/pierceboggan/status/2067449111809597803))
4. **最强的构建者模式，是围绕智能体做中间件：记忆、验证、secrets 和设计系统。** AI DevKit、keymaxxer、Open Design、gstack 和 ClaudeKit 各自都在填补基础智能体仍未补上的工作流缺口。([source](https://x.com/DanKornas/status/2067593408697311734))
